## 1. บทสรุปผู้บริหาร (Executive Summary)

การทบทวนนี้ประเมินความสอดคล้องของสถาปัตยกรรมโครงการ PKA ณ จุดที่ตระกูลแนวคิด (STD, ARC, DOM, REF) และตระกูลปฏิบัติการ (OPS-001 ถึง OPS-005A) จัดทำเสร็จแล้ว แต่ยัง **ไม่เริ่ม** ชั้นการปฏิบัติ (Workflow, Event, Agent, Worker) — จึงเป็นจังหวะที่เหมาะสมที่สุดในการตรวจสถาปัตยกรรมฐานก่อนเดินหน้าต่อ

### 1.1 ขอบเขตและวิธีการประเมิน

- **ขอบเขต:** ตระกูล STD, ARC, DOM, REF, OPS เท่านั้น — ไม่ตรวจชั้นการปฏิบัติและเทคโนโลยี
- **วิธีการ:** ตรวจสิบมิติ — บูรณภาพของชั้น, ขอบเขต, การสืบย้อน, การพึ่งพา, ศัพท์, ความซ้ำซ้อน, ความครบถ้วน, การเกาะเกี่ยว/ยึดเหนี่ยว, ธรรมาภิบาล แล้วให้คะแนนรวมและจัดกลุ่มช่องว่าง

### 1.2 ภาพรวมสุขภาพสถาปัตยกรรม (Architecture Health)

> **สุขภาพโดยรวม: แข็งแรง (Healthy) — คะแนนรวม 80 / 100**
> สถาปัตยกรรม PKA มีรากฐานที่มั่นคง ลำดับชั้นชัดเจน การพึ่งพามีทิศทางเดียวไม่มีวงวน และหลักธรรมาภิบาล (โดยเฉพาะข้อจำกัดของผู้ช่วยอัจฉริยะ) สอดคล้องกันทั้งระบบ ช่องว่างที่พบเป็นระดับ **High ลงมา ไม่มีปัญหาระดับ Critical** และทั้งหมดแก้ไขได้ก่อนเข้าสู่ชั้นการปฏิบัติ

### 1.3 สรุประดับประเด็น (Issue Summary)

| ระดับ | จำนวน | สาระสำคัญ |
|---|---|---|
| **Critical** | 0 | ไม่พบปัญหาที่ขวางการเดินหน้า |
| **High** | 2 | (H1) สะพานสืบย้อนระหว่างแบบจำลองโดเมน (DOM) กับขีดความสามารถ/หน้าที่ (OPS) ยังไม่ชัด; (H2) ข้อมูลสำนักงานซ้ำกันระหว่าง OPS-001 §10 กับ OPS-004 |
| **Medium** | 5 | แผนผังลำดับชั้นยังไม่รวม OPS/REV; ศัพท์ "หน้าที่" vs "หน้าที่ทางธุรกิจ"; แหล่งนิยามศัพท์ซ้ำ (STD-001 §24 vs REF-001); ธรรมาภิบาล STD-008 vs OPS; เส้นแบ่ง ARC-004 vs DOM-003 |
| **Low** | 3 | ช่องว่างเลขลำดับ STD-009; รอบการทบทวน REV ยังไม่กำหนด; ตระกูล ADR ยังว่าง |

> **ข้อสรุปสำหรับผู้บริหาร:** สถาปัตยกรรมพร้อมเข้าสู่ **OPS-006 Workflow Architecture** ได้ โดยควรปิดประเด็น High สองข้อก่อน เพื่อไม่ต้องย้อนกลับมาแก้สถาปัตยกรรมหลักภายหลัง

## 2. การทบทวนบูรณภาพของชั้น (Layer Integrity Review)

ตรวจว่าแต่ละตระกูลมีขอบเขตชัดเจน ไม่ทับซ้อน และไม่มีชั้นใดทำผิดหน้าที่

| ตระกูล | บทบาทที่ประกาศไว้ | บูรณภาพ |
|---|---|---|
| **STD** | กฎและหลักการกลาง (Vision + Standards) | ดี — รักษาบทบาท "กฎ" |
| **ARC** | ภาพรวมของระบบและการเชื่อมโยง | ดี — ไม่ลงรายละเอียดกฎหรือแบบจำลอง |
| **DOM** | แบบจำลองเชิงแนวคิดขององค์ความรู้ | ดี — แยกจากองค์ความรู้จริง (KNW) ชัด |
| **REF** | คำอธิบายแหล่งอ้างอิงต้นทาง | ดี — ไม่กำหนดกฎหรือแบบจำลอง |
| **OPS** | สถาปัตยกรรมองค์กรและธุรกิจ (Operations) | ดี — แต่ยังไม่ปรากฏในแผนผังลำดับชั้นหลัก |

**ข้อสังเกต**

- ตระกูล OPS เป็น **ชั้นปฏิบัติการ/สถาปัตยกรรมองค์กร** ที่เพิ่มเข้ามาหลังการวางลำดับชั้น 7 ระดับเดิม แผนผังลำดับชั้นในพอร์ทัล (หัวข้อ 3) จึง **ยังไม่แสดงตำแหน่งของ OPS และ REV** อย่างชัดเจน → ประเด็น **Medium (M1)**
- ภายในตระกูล OPS การแบ่งชั้นย่อยชัดเจนมาก: Charter → Organization → Capability → Office → Assignment → Business Function โดยแต่ละฉบับถือหนึ่งความรับผิดชอบ — บูรณภาพของชั้นภายใน OPS อยู่ในระดับสูง

> **สรุป:** บูรณภาพของชั้นอยู่ในเกณฑ์ดี ไม่มีชั้นใดทำผิดหน้าที่ ข้อบกพร่องเดียวคือแผนผังลำดับชั้นหลักยังไม่อัปเดตให้รวมชั้น OPS และ REV ในฐานะชั้นขวางระดับ

## 3. การทบทวนขอบเขต (Boundary Review)

ตรวจว่าแต่ละตระกูลเป็นสิ่งที่ตนอ้างจริง และไม่มีตระกูลใดทำหน้าที่แทนอีกตระกูล

| คำถาม | ผล | หมายเหตุ |
|---|---|---|
| STD เป็น "มาตรฐาน" จริงหรือไม่ | ✔ ใช่ | กำหนดกฎ ไม่ลงรายละเอียดภาพรวมหรือแบบจำลอง |
| ARC เป็น "สถาปัตยกรรม" จริงหรือไม่ | ✔ ใช่ | อธิบายภาพรวม ไม่กำหนดกฎเอง |
| DOM เป็น "แบบจำลองโดเมน" จริงหรือไม่ | ✔ ใช่ | แบบจำลองเชิงแนวคิด ไม่ใช่องค์ความรู้จริง |
| REF เป็น "แหล่งอ้างอิง" จริงหรือไม่ | ✔ ใช่ | อธิบายคัมภีร์/แหล่งอ้างอิง |
| OPS เป็น "สถาปัตยกรรมองค์กร" จริงหรือไม่ | ✔ ใช่ (มีข้อสังเกต) | เป็นสถาปัตยกรรมองค์กร+ธุรกิจ; เส้นแบ่งกับ ARC ต้องเฝ้าระวัง |

**ข้อสังเกตเรื่องเส้นแบ่ง**

- **ธรรมาภิบาล:** [STD-008](../standards/PKA-STD-008-มาตรฐานการพัฒนาและการกำกับเอกสาร.md) กำหนด *กฎ* ด้านการกำกับเอกสาร ขณะที่ OPS-001/OPS-005 กำหนด *การนำกฎไปใช้เชิงองค์กร* — เส้นแบ่งถูกต้องในหลักการ แต่มีความเสี่ยงทับซ้อนหากไม่ระบุชัดว่า "กฎ" อยู่ที่ STD และ "การประยุกต์" อยู่ที่ OPS → ประเด็น **Medium (M4)**
- **เชิงความหมาย:** [ARC-004 สถาปัตยกรรมเชิงความหมาย](../architecture/PKA-ARC-004-สถาปัตยกรรมเชิงความหมาย.md) อธิบาย *แนวทาง* เชิงความหมายระดับภาพรวม ขณะที่ [DOM-003 แค็ตตาล็อกชนิดความสัมพันธ์](../domain/PKA-DOM-003-แค็ตตาล็อกชนิดความสัมพันธ์ขององค์ความรู้.md) กำหนด *ชนิดความสัมพันธ์* ระดับแบบจำลอง — เส้นแบ่งต้องชัดว่า "ภาพรวม=ARC, ชนิด=DOM" → ประเด็น **Medium (M5)**

> **สรุป:** ทุกตระกูลอยู่ในขอบเขตของตน ไม่มีตระกูลใดทำหน้าที่แทนอีกตระกูล เส้นแบ่งสองจุด (ธรรมาภิบาล, เชิงความหมาย) ควรระบุเจ้าของให้คมขึ้นเพื่อกันการทับซ้อนในอนาคต

## 4. การทบทวนการสืบย้อน (Traceability Review)

ตรวจสายการสืบย้อนตั้งแต่วิสัยทัศน์จนถึงชั้นการปฏิบัติที่วางแผนไว้

```
   Vision (STD-001)
        ▼
   Standards (STD-002+)
        ▼
   Architecture (ARC-*)
        ▼
   Domain (DOM-*)
        ▼
   Reference (REF-*)
        ▼  ── สะพานที่ยังไม่ชัด (H1) ──
   Capability (OPS-003)
        ▼
   Office (OPS-004)
        ▼
   Assignment (OPS-005)
        ▼
   Business Function (OPS-005A)
        ▼
   Workflow (OPS-006, Planned)
        ▼
   Event (OPS-007, Planned)
        ▼
   Agent (OPS-008, Planned)
        ▼
   Worker (OPS-009, Planned)
```

**ผลการตรวจ**

- **สายแนวคิด (STD→ARC→DOM→REF):** สืบย้อนได้ครบ — ทุกฉบับอ้างอิงเอกสารแม่ STD-001 และระบุเอกสารระดับบนที่ขยายความ
- **สายปฏิบัติการ (OPS-003→005A→...):** สืบย้อนได้ครบและชัดเจน — OPS-005A เติมข้อต่อ Business Function ที่เคยขาดระหว่างขีดความสามารถกับกระแสงาน ทำให้สาย Capability → Function → Workflow → Event → Agent → Worker ต่อเนื่อง
- **จุดขาด (H1):** สะพานระหว่าง **สายแนวคิด** กับ **สายปฏิบัติการ** ยังไม่ชัด — ขีดความสามารถ/หน้าที่ใน OPS (เช่น Knowledge Modeling, Relationship Management) ควรสืบย้อนกลับไปยัง **แบบจำลองโดเมน (DOM)** ได้อย่างเป็นทางการ แต่ปัจจุบัน OPS อ้างถึง STD-001/ARC-001 เป็นหลัก ส่วนการเชื่อมกับ DOM ยังเป็นนัย ไม่ใช่การสืบย้อนที่ระบุชัด → **High (H1)**

> **สรุป:** การสืบย้อนภายในแต่ละสายแข็งแรง แต่ **รอยต่อระหว่างสายแนวคิด (DOM) กับสายปฏิบัติการ (OPS Capability/Function)** เป็นจุดที่ต้องเสริมก่อนสร้างชั้น Workflow

## 5. การทบทวนการพึ่งพา (Dependency Review)

| มิติ | ผล |
|---|---|
| **ทิศทางการพึ่งพา (Direction)** | สม่ำเสมอ — เอกสารระดับล่างอ้างอิงระดับบน (OPS → ARC → STD) ไม่มีทิศทางย้อนกลับ |
| **การพึ่งพาแบบวงวน (Circular)** | **ไม่พบ** — ไม่มีกรณี เช่น OPS อ้าง DOM แล้ว DOM อ้าง OPS กลับ |
| **การพึ่งพาขาด (Broken)** | ไม่พบลิงก์เสียในสายอ้างอิงหลัก |
| **ความสอดคล้องของทิศทาง** | ดี — REV (ฉบับนี้) อ้างอิงข้ามทุกระดับได้ตามสถานะ "ขวางทุกระดับ" |

**ข้อสังเกต**

- ตระกูลแนวคิด (STD/ARC/DOM/REF) **ไม่อ้างอิงลงไปยัง OPS** ซึ่งถูกต้อง — รักษาทิศทางบนลงล่าง
- ภายใน OPS การพึ่งพาเรียงเป็นสายเดียว (001→002→003→004→005→005A) ไม่มีการอ้างวกกลับ
- เอกสาร REV พึ่งพาเอกสารจำนวนมาก (โดยธรรมชาติของงานตรวจ) — เป็นการเกาะเกี่ยวที่ยอมรับได้สำหรับเอกสารประเมิน

> **สรุป:** มิติการพึ่งพาเป็น **จุดแข็งที่สุด** ของสถาปัตยกรรม — ทิศทางเดียว ไม่มีวงวน ไม่มีลิงก์ขาดในสายหลัก

## 6. การทบทวนความสอดคล้องของศัพท์ (Terminology Consistency Review)

| ศัพท์ | ไทย | ความสอดคล้อง |
|---|---|---|
| Capability | ขีดความสามารถ | สม่ำเสมอทั้ง OPS |
| Business Function | หน้าที่ทางธุรกิจ | สม่ำเสมอ (หลังแก้ OPS-003 ให้ตรง OPS-005A) |
| Workflow | กระแสงาน | สม่ำเสมอ |
| Office | สำนักงาน | สม่ำเสมอ |
| Agent | ตัวแทนงาน | สม่ำเสมอ |
| Worker | หน่วยปฏิบัติ | สม่ำเสมอ |
| Relationship | ความสัมพันธ์ | สม่ำเสมอ (DOM-003 ↔ OPS) |
| Citation | การอ้างอิง | สม่ำเสมอ (STD-003 ↔ OPS) |
| Reference | แหล่งอ้างอิง | สม่ำเสมอ |
| Semantic | เชิงความหมาย | สม่ำเสมอ |
| Function / Activity / Task | หน้าที่ / กิจกรรม / งาน | มีความเสี่ยง (ดูล่าง) |

**ข้อสังเกต**

- **"หน้าที่" กับ "หน้าที่ทางธุรกิจ":** OPS-001/OPS-002 ใช้คำ "หน้าที่" ในความหมายของ *ภารกิจสำนักงาน* ขณะที่ OPS-005A ใช้ "หน้าที่ทางธุรกิจ (Business Function)" เป็นชั้นสถาปัตยกรรม — คำว่า "หน้าที่" ลอย ๆ จึงมีสองความหมาย ควรกำหนดให้ "หน้าที่ทางธุรกิจ" เป็นศัพท์เฉพาะเสมอ → **Medium (M2)**
- **แหล่งนิยามศัพท์:** [REF-001 อภิธานศัพท์มาตรฐาน](../reference/PKA-REF-001-อภิธานศัพท์มาตรฐานของโครงการ.md) ควรเป็น **แหล่งความจริงเดียว (Source of Truth)** ของนิยามศัพท์ แต่ STD-001 §24 ก็มีอภิธานศัพท์ของตน → ดูหัวข้อ 7 (M3)

> **สรุป:** ศัพท์หลักสอดคล้องกันดีทั้งระบบ จุดเสี่ยงคือคำว่า "หน้าที่" ที่มีสองความหมาย และการมีอภิธานศัพท์สองแหล่ง

## 7. การทบทวนความซ้ำซ้อน (Duplication Review)

| คู่ที่ตรวจ | ผล | แหล่งความจริงเดียวที่เสนอ |
|---|---|---|
| **OPS-003 (Capability) vs OPS-005A (Business Function)** | แยกชัด — "ทำได้" vs "ต้องทำ" ไม่ซ้ำ | Capability = OPS-003; Function = OPS-005A |
| **OPS-001 §10 (Office) vs OPS-004 (Office Catalog)** | **ซ้ำ** — ภารกิจ/ขอบเขต/ปัจจัยนำเข้า/ผลผลิตของสำนักงานปรากฏทั้งสองที่ | **OPS-004 เป็น Source of Truth**; OPS-001 ควรเหลือเพียงภาพรวม → **High (H2)** |
| **STD-001 §24 (อภิธานศัพท์) vs REF-001 (Canonical Glossary)** | ซ้ำบางส่วน | **REF-001 เป็น Source of Truth** ของนิยามศัพท์ → **Medium (M3)** |
| **ARC-004 (เชิงความหมาย) vs DOM-003 (ชนิดความสัมพันธ์)** | คาบเกี่ยว | ภาพรวม=ARC-004; ชนิด=DOM-003 → **Medium (M5)** |

**ข้อสังเกต**

- ความซ้ำที่ชัดที่สุดคือ **ข้อมูลสำนักงานใน OPS-001 §10 กับ OPS-004** — ทั้งสองให้ภารกิจ/ขอบเขต/ปัจจัยนำเข้า/ผลผลิตของ 11 สำนักงาน หากปล่อยไว้จะเกิดความเสี่ยงที่เนื้อหาทั้งสองจะคลาดเคลื่อนเมื่อแก้ฝั่งเดียว → ควรกำหนด OPS-004 เป็นเจ้าของเนื้อหา และให้ OPS-001 คงเพียงรายชื่อและบทบาทระดับสูง

> **สรุป:** ความซ้ำส่วนใหญ่อยู่ในระดับจัดการได้ ยกเว้นคู่ OPS-001 §10 ↔ OPS-004 ที่ควรกำหนดแหล่งความจริงเดียวโดยเร็ว (H2)

## 8. การทบทวนความครบถ้วนของสถาปัตยกรรม (Architecture Completeness Review)

ประเมินว่าสถาปัตยกรรมยังขาดชั้นใดที่เป็นมาตรฐานของสถาปัตยกรรมองค์กร — **เสนอเท่านั้น ยังไม่สร้างเอกสาร**

| ชั้นที่อาจขาด | สถานะปัจจุบัน | ความจำเป็น |
|---|---|---|
| **Information Object (วัตถุสารสนเทศ)** | DOM กำหนดแบบจำลอง แต่ไม่มีแค็ตตาล็อกวัตถุสารสนเทศที่เชื่อม DOM กับขีดความสามารถ OPS | สูง — ช่วยอุดสะพาน DOM↔OPS (H1) |
| **Business Role (บทบาททางธุรกิจ)** | OPS-002 มีหน่วยองค์กร (สำนักงาน/ตัวแทนงาน/หน่วยปฏิบัติ) แต่ไม่มีแค็ตตาล็อก *บทบาท* แยก | ปานกลาง |
| **Policy (นโยบาย)** | หลักนโยบายกระจายใน STD-001 และ OPS-001 | ปานกลาง — ควรรวมเป็นแค็ตตาล็อกนโยบาย |
| **Business Rule (กฎทางธุรกิจ)** | กฎฝังอยู่ใน STD | ต่ำ–ปานกลาง |
| **Service (บริการ)** | ยังไม่มี (เป็นเรื่องชั้นการปฏิบัติ) | ต่ำ — เหมาะกับเฟสหลัง |
| **Decision Model (แบบจำลองการตัดสินใจ)** | ตระกูล ADR สงวนไว้แต่ยังว่าง | ปานกลาง |

> **สรุป:** สถาปัตยกรรมครบพอสำหรับเข้าสู่ชั้นการปฏิบัติ แต่ชั้น **Information Object** มีคุณค่าสูงสุดที่ควรพิจารณาก่อน เพราะช่วยอุดรอยต่อ DOM↔OPS (H1) — ส่วนชั้นอื่นเสนอไว้พิจารณาในเฟสถัดไป โดย **ยังไม่สร้างเอกสารในรอบนี้**

## 9. การทบทวนการเกาะเกี่ยวและการยึดเหนี่ยว (Coupling & Cohesion Review)

| มิติ | การประเมิน |
|---|---|
| **การยึดเหนี่ยวภายในเอกสาร (Cohesion)** | สูง — แต่ละฉบับใน OPS ถือหนึ่งความรับผิดชอบชัดเจน |
| **การเกาะเกี่ยวระหว่างเอกสาร (Coupling)** | เหมาะสม — อ้างอิงผ่านลิงก์ที่ระบุหัวข้อ ไม่ผูกแน่นเกินไป |
| **ความหนาแน่นของการอ้างอิงข้าม (Cross-Reference Density)** | สูงภายใน OPS (ดี); ต่ำระหว่าง DOM↔OPS (ช่องว่าง H1) |
| **ความหนาแน่นของการพึ่งพา (Dependency Density)** | ต่ำและมีทิศทางเดียว (ดี) |

**ข้อเสนอเชิงโครงสร้าง**

- **ควรเพิ่มการยึดเหนี่ยวระหว่าง DOM กับ OPS** — ปัจจุบันสองสายนี้เกาะเกี่ยวกันน้อยเกินไป ทำให้ขีดความสามารถด้านองค์ความรู้ไม่ผูกกับแบบจำลองโดเมนอย่างเป็นทางการ
- **ควรลดการเกาะเกี่ยวเชิงเนื้อหาซ้ำระหว่าง OPS-001 กับ OPS-004** — เปลี่ยนจากการทำสำเนาเนื้อหา (content coupling) เป็นการอ้างอิง (reference coupling) โดยให้ OPS-004 เป็นเจ้าของ

> **สรุป:** การยึดเหนี่ยวภายในสูงและการพึ่งพ่าต่ำเป็นจุดแข็ง จุดที่ต้องปรับคือเพิ่มการเชื่อม DOM↔OPS และลดสำเนาเนื้อหา OPS-001↔OPS-004

## 10. การทบทวนธรรมาภิบาล (Governance Review)

ตรวจว่าบทบาทธรรมาภิบาลแยกจากกันชัดเจน

| บทบาท | เจ้าภาพ | แยกชัดหรือไม่ |
|---|---|---|
| **อำนาจสูงสุด (Authority)** | ผู้อำนวยการสูงสุด / มนุษย์ | ✔ ชัด (OPS-002, OPS-005 §7) |
| **การอนุมัติ (Approval)** | มนุษย์เท่านั้นสำหรับองค์ความรู้ | ✔ ชัด (STD-001 §13, OPS-001 §19, OPS-005 §7) |
| **ความเป็นเจ้าของ (Ownership)** | สำนักงานเจ้าของ | ✔ ชัด (OPS-004, OPS-005) |
| **การทบทวน (Review)** | สำนักคุณภาพ + มนุษย์ผู้ทบทวน | ✔ ชัด (OPS-005 §9) |
| **การตรวจสอบ (Audit)** | ตระกูล REV (ฉบับนี้) | ✔ มี แต่ยังไม่กำหนดรอบ |

**ผลสำคัญ**

- **ผู้ช่วยอัจฉริยะ (AI) ไม่มีสิทธิ์อนุมัติ และไม่มีสิทธิ์เป็นแหล่งความจริง (Source of Truth)** — หลักนี้สอดคล้องกันทั้งระบบ ตั้งแต่ STD-001 §10/§13 ถึง OPS-001 §19–20 และ OPS-005 §7 ถือเป็น **จุดแข็งด้านธรรมาภิบาลที่โดดเด่นที่สุด**
- บทบาท เจ้าของ / ผู้อนุมัติ / ผู้ทบทวน แยกจากกันชัดเจนใน OPS-005 — ไม่มีบทบาทใดควบรวมจนเสียการถ่วงดุล
- **ข้อสังเกตเล็กน้อย:** ตระกูล REV ทำหน้าที่ตรวจสอบ แต่ยังไม่กำหนด *รอบ* หรือ *ตัวกระตุ้น* ของการทบทวน → **Low (L2)**

> **สรุป:** ธรรมาภิบาลเป็นจุดแข็ง การแยกบทบาทชัดเจน และข้อจำกัดของผู้ช่วยอัจฉริยะถูกบังคับใช้สอดคล้องกันทั้งระบบ

## 13. ข้อเสนอแนะ (Recommendations)

### 13.1 ระยะสั้น (ก่อนเริ่ม OPS-006)

1. **ปิด H2** — กำหนด OPS-004 เป็น Source of Truth ของข้อมูลสำนักงาน และลด OPS-001 §10 เหลือภาพรวม (ลำดับ 1)
2. **ปิด M1** — อัปเดตแผนผังลำดับชั้นในพอร์ทัลให้รวม OPS (ชั้นปฏิบัติการ) และ REV (ขวางทุกระดับ)
3. **ปิด M3** — ประกาศ REF-001 เป็นแหล่งความจริงเดียวของศัพท์ และให้อภิธานศัพท์อื่นอ้างอิงกลับ
4. **ปิด M2** — ใช้ "หน้าที่ทางธุรกิจ" เป็นศัพท์เฉพาะอย่างสม่ำเสมอ

### 13.2 ระยะกลาง (ระหว่างพัฒนา OPS-006/007)

5. **ปิด H1** — สร้างการจับคู่ DOM ↔ ขีดความสามารถ/หน้าที่ทางธุรกิจอย่างเป็นทางการ (พิจารณาชั้น Information Object)
6. **ปิด M4, M5** — ระบุเจ้าของเนื้อหาด้านธรรมาภิบาล (STD vs OPS) และเชิงความหมาย (ARC vs DOM) ให้คมขึ้น
7. กำหนดรอบการทบทวน REV (L2) และเริ่มบันทึก ADR (L3)

### 13.3 ระยะยาว (พร้อมเฟสการปฏิบัติและองค์ความรู้)

8. พิจารณาชั้นสถาปัตยกรรมเพิ่มเติมตามความจำเป็น — Information Object, Business Role, Policy, Decision Model — โดยไม่สร้างก่อนถึงเวลา
9. รักษาวินัยการสืบย้อนและแหล่งความจริงเดียวเมื่อจำนวนเอกสารเพิ่มขึ้น

## 14. ข้อเสนอแนะด้าน Roadmap (Roadmap Recommendation)

**คำถาม:** หลังการทบทวนนี้ ควรทำอะไรต่อ — เริ่ม **OPS-006 Workflow Architecture** หรือเริ่ม **PKA-KNW-0001 องค์ความรู้รายการแรก**?

**ข้อเสนอ:** เดินหน้าสู่ **OPS-006 Workflow Architecture** เป็นลำดับถัดไป โดยปิดประเด็น High สองข้อก่อน

| ทางเลือก | ความพร้อม | เหตุผล |
|---|---|---|
| **OPS-006 Workflow Architecture** | ✔ พร้อม (แนะนำ) | สาย Capability → Function ครบแล้ว กระแสงานคือข้อต่อถัดไปตามธรรมชาติ |
| **PKA-KNW-0001 (องค์ความรู้รายการแรก)** | ✖ ยังไม่ควร | องค์ความรู้จริงต้องพึ่งแบบจำลองโดเมน (DOM) + กระแสงาน (OPS-006) ที่ยังไม่นิยาม การเริ่มก่อนจะต้องย้อนแก้ภายหลัง |

**ลำดับที่แนะนำ**

```
   1) ปิด H1 (สะพาน DOM↔OPS) + H2 (Source of Truth สำนักงาน)
        ▼
   2) OPS-006 Workflow Architecture
        ▼
   3) OPS-007 Event Architecture
        ▼
   4) OPS-008 / OPS-009 Agent & Worker Architecture
        ▼
   5) PKA-KNW-0001 องค์ความรู้รายการแรก (เมื่อสายการปฏิบัติพร้อม)
```

> **ข้อสรุป:** สถาปัตยกรรมฐาน **แข็งแรงพอ (80/100)** ที่จะเข้าสู่ชั้น Workflow โดยไม่ต้องย้อนกลับมาเปลี่ยนสถาปัตยกรรมหลัก — ขอเพียงปิดประเด็น High สองข้อก่อน เพื่อให้กระแสงานสร้างบนรากฐานที่ผูกกับแบบจำลองโดเมนและมีแหล่งความจริงเดียว องค์ความรู้จริง (KNW) ควรรอจนชั้นการปฏิบัติพร้อม
