> **หมายเหตุสำคัญ**
> เอกสารฉบับนี้นิยาม **หน้าที่ทางธุรกิจ (Business Function)** ซึ่งเป็น **"หน้าที่ถาวรขององค์กร"** และเป็น **ชั้นสถาปัตยกรรมทางธุรกิจ (Business Architecture Layer)** ที่ขาดหายไประหว่างขีดความสามารถ (Capability) กับกระแสงาน (Workflow)
> หน้าที่ทางธุรกิจตอบคำถาม **"องค์กรมีหน้าที่ถาวรอะไร"** — **ไม่ใช่** กระแสงาน (Workflow) **ไม่ใช่** กระบวนการ (Process) **ไม่ใช่** งาน (Task) **ไม่ใช่** กิจกรรม (Activity) และ **ไม่ใช่** ตัวแทนงาน (Agent)
> เอกสารฉบับนี้เป็น **สถาปัตยกรรมเชิงแนวคิด (Conceptual Architecture)** เท่านั้น **ไม่กล่าวถึงเทคโนโลยี เครื่องมือ วิธีพัฒนา หรือรายละเอียดการปฏิบัติใด ๆ** และต้องสอดคล้องกับ **[PKA-STD-001](../standards/PKA-STD-001-ธรรมนูญโครงการ.md) (เอกสารแม่)** และเอกสาร OPS-003 ถึง OPS-005 เสมอ

## 1. วัตถุประสงค์ (Purpose)

เอกสารฉบับนี้กำหนด **สารบบหน้าที่ทางธุรกิจ (Business Function Catalog)** ของสำนักงานองค์ความรู้ดิจิทัล โดยเติมชั้นที่ขาดหายไประหว่าง **ขีดความสามารถ (Capability)** ใน [PKA-OPS-003](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md) กับ **กระแสงาน (Workflow)** ที่จะออกแบบใน PKA-OPS-006

เอกสารนี้มีไว้เพื่อ

1. นิยาม **หน้าที่ทางธุรกิจ** ในฐานะ "หน้าที่ถาวรขององค์กร" ที่คงทนข้ามการเปลี่ยนแปลงของผู้ทำและวิธีทำ
2. วาง **ชั้นสถาปัตยกรรมทางธุรกิจ (Business Architecture Layer)** ให้เป็นสะพานเชื่อม **สถาปัตยกรรมขีดความสามารถ (Capability Architecture)** กับ **สถาปัตยกรรมการปฏิบัติ (Execution Architecture)**
3. กำหนด **ลำดับชั้น การเป็นเจ้าของ การพึ่งพา การจำแนกประเภท วงจรชีวิต และธรรมาภิบาล** ของหน้าที่ทางธุรกิจ
4. ทำให้สายการสืบย้อนของตระกูล OPS สมบูรณ์: **ขีดความสามารถ → หน้าที่ทางธุรกิจ → กระแสงาน → เหตุการณ์ → ตัวแทนงาน → หน่วยปฏิบัติ**

> เมื่ออ่านเอกสารฉบับนี้จบ ผู้อ่านต้องเข้าใจว่า **หน้าที่ทางธุรกิจคือ "หน้าที่ถาวรขององค์กร"** ซึ่งเป็นตัวเชื่อมระหว่างขีดความสามารถกับกระแสงาน และเป็นรากฐานของกระแสงาน เหตุการณ์ ตัวแทนงาน และหน่วยปฏิบัติ โดย **ไม่ผูกกับบุคคล ไม่ผูกกับผู้ช่วยอัจฉริยะ ไม่ผูกกับเทคโนโลยี และไม่ผูกกับวิธีการพัฒนา**

## 2. ขอบเขต (Scope)

**อยู่ในขอบเขต**

- นิยาม คุณสมบัติ และหลักการของหน้าที่ทางธุรกิจ
- สารบบหน้าที่ทางธุรกิจระดับสูงขององค์กร พร้อมขีดความสามารถต้นทางและสำนักงานเจ้าของ
- ลำดับชั้น การเป็นเจ้าของ การพึ่งพา การจำแนกประเภท วงจรชีวิต ธรรมาภิบาล และคุณภาพของหน้าที่
- การสืบย้อนเชื่อมหน้าที่ทางธุรกิจเข้ากับขีดความสามารถและกระแสงาน

**อยู่นอกขอบเขต**

- กระแสงานจริง (อยู่ใน **PKA-OPS-006**)
- เหตุการณ์จริง (อยู่ใน **PKA-OPS-007**)
- ตัวแทนงานจริง (อยู่ใน **PKA-OPS-008**)
- หน่วยปฏิบัติจริง (อยู่ใน **PKA-OPS-009**)
- กระบวนการ งาน กิจกรรม การทำงานอัตโนมัติ และรายละเอียดการปฏิบัติทั้งหมด
- เครื่องมือ เทคโนโลยี วิธีพัฒนา หรือรายละเอียดเชิงเทคนิคใด ๆ

> เอกสารฉบับนี้ตอบว่า **"องค์กรมีหน้าที่ถาวรอะไร"** เท่านั้น ส่วนคำถามว่าหน้าที่นั้น *ถูกทำผ่านลำดับงานใด* เป็นของ OPS-006 เป็นต้นไป

## 3. ความหมายของหน้าที่ทางธุรกิจ (What is a Business Function)

**หน้าที่ทางธุรกิจ (Business Function)** คือ *หน้าที่ถาวร* ที่องค์กรต้องกระทำอย่างต่อเนื่องเพื่อให้ขีดความสามารถของตนเกิดผลจริง หน้าที่ทางธุรกิจเป็นสิ่งที่ **คงอยู่** ไม่ว่าผู้ทำจะเป็นใคร ใช้วิธีใด หรือผ่านลำดับงานแบบใด

หน้าที่ทางธุรกิจมีลักษณะสำคัญสามประการ

- **ถาวร** — หน้าที่ทางธุรกิจไม่ใช่กิจกรรมที่เกิดขึ้นชั่วคราว แต่เป็นหน้าที่ที่องค์กรต้องทำตราบที่องค์กรยังดำรงอยู่
- **เป็นนามธรรมเหนือวิธีทำ** — หน้าที่ทางธุรกิจบอกว่า *องค์กรทำหน้าที่อะไร* ไม่ใช่ *ทำอย่างไร* วิธีทำเป็นเรื่องของกระแสงาน
- **เป็นสะพาน** — หน้าที่ทางธุรกิจอยู่กึ่งกลางระหว่างขีดความสามารถ (สิ่งที่องค์กรทำได้) กับกระแสงาน (วิธีที่งานไหล) จึงเป็นข้อต่อที่ทำให้สองชั้นนี้เชื่อมกันได้

> หน้าที่ทางธุรกิจเป็นมาตรฐานของสถาปัตยกรรมองค์กร (Enterprise Architecture) ที่ช่วยป้องกันไม่ให้องค์กรกระโดดจาก "ความสามารถ" ไปสู่ "วิธีทำ" โดยข้ามชั้น "หน้าที่" ที่ควรมั่นคงและถาวร

## 4. หลักการของหน้าที่ทางธุรกิจ (Business Function Principle)

1. **หน้าที่ถาวร ไม่ใช่กิจกรรมชั่วคราว** — หน้าที่ทางธุรกิจคงอยู่ข้ามเวลา ส่วนกิจกรรมเกิดขึ้นและจบไป
2. **หน้าที่ผูกกับองค์กร ไม่ผูกกับผู้ทำ** — หน้าที่ทางธุรกิจเป็นขององค์กรผ่านสำนักงานเจ้าของ ไม่ใช่ของบุคคลหรือผู้ช่วยอัจฉริยะ
3. **หนึ่งหน้าที่ สืบมาจากขีดความสามารถ** — ทุกหน้าที่ทางธุรกิจต้องสืบย้อนไปยังขีดความสามารถต้นทางใน OPS-003 ได้
4. **หน้าที่รองรับกระแสงานได้หลายแบบ** — หน้าที่ทางธุรกิจหนึ่งอาจถูกทำผ่านกระแสงานได้หลายรูปแบบ โดยตัวหน้าที่ไม่เปลี่ยน
5. **หน้าที่ไม่ทับซ้อนและไม่มีช่องว่าง** — หน้าที่ทางธุรกิจทั้งหมดรวมกันต้องครอบคลุมขีดความสามารถขององค์กรโดยไม่ซ้ำซ้อน

## 5. ความแตกต่างระหว่างแนวคิด (Capability vs Function vs Workflow vs Task vs Activity)

| แนวคิด | ตอบคำถาม | ความคงทน |
|---|---|---|
| **ขีดความสามารถ (Capability)** | องค์กร *ทำอะไรได้* | สูงสุด — เปลี่ยนยากที่สุด |
| **หน้าที่ทางธุรกิจ (Business Function)** | องค์กรมี *หน้าที่ถาวรอะไร* | สูง — ถาวรแต่ขยายได้ |
| **กระแสงาน (Workflow)** | งาน *ไหลอย่างไร* | ปานกลาง — เปลี่ยนได้ตามวิธีทำ |
| **งาน (Task)** | ต้อง *ทำสิ่งใดให้เสร็จ* | ต่ำ — เกิดและจบเป็นรายการ |
| **กิจกรรม (Activity)** | *กำลังลงมือทำสิ่งใด* | ต่ำสุด — ชั่วคราวที่สุด |

ความสัมพันธ์เชิงลำดับ

- **ขีดความสามารถ** กำหนดว่าองค์กรทำอะไรได้ → **หน้าที่ทางธุรกิจ** แปรความสามารถนั้นเป็นหน้าที่ถาวร → **กระแสงาน** กำหนดวิธีที่หน้าที่นั้นถูกทำ → **งาน** และ **กิจกรรม** คือการลงมือจริงในกระแสงาน
- หน้าที่ทางธุรกิจอยู่ **เหนือ** กระแสงาน งาน และกิจกรรม แต่ **ใต้** ขีดความสามารถ จึงเป็นสะพานที่ทั้งมั่นคงและเชื่อมต่อได้

> ความต่างที่เป็นหัวใจ: **ขีดความสามารถคือ "ทำได้" หน้าที่ทางธุรกิจคือ "ต้องทำ" กระแสงานคือ "ทำอย่างไร"** — เอกสารฉบับนี้รับผิดชอบเฉพาะชั้น "ต้องทำ"

## 6. ลำดับชั้นของหน้าที่ (Function Hierarchy)

หน้าที่ทางธุรกิจวางตัวอยู่ในลำดับชั้นที่เชื่อมขีดความสามารถลงสู่การปฏิบัติ

```
            ┌─────────────────────────────────────┐
            │           Capability                │  ← องค์กรทำอะไรได้ (OPS-003)
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │        Business Function            │  ← หน้าที่ถาวรขององค์กร (เอกสารนี้)
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │           Sub Function              │  ← หน้าที่ย่อยภายในหน้าที่
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │            Workflow                 │  ← วิธีที่หน้าที่ถูกทำ (OPS-006)
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │             Event                   │  ← การประสานผ่านเหตุการณ์ (OPS-007)
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │             Agent                   │  ← ผู้รับผิดชอบภารกิจ (OPS-008)
            └─────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────┐
            │             Worker                  │  ← ผู้ลงมือทำงานหนึ่งหน่วย (OPS-009)
            └─────────────────────────────────────┘
```

> ลำดับชั้นนี้ขยายความจาก [OPS-003 หัวข้อ 5](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#5-ลำดับชั้นของขีดความสามารถ-capability-hierarchy) ซึ่งระบุว่า **หน้าที่ทางธุรกิจ (Business Function)** เป็นชั้นหนึ่งในลำดับชั้นขีดความสามารถ และรายละเอียดของชั้นนี้ถูกกำหนดไว้ในเอกสารฉบับนี้ เพื่อรักษาการแยกหน้าที่ของเอกสาร

## 7. สารบบหน้าที่ทางธุรกิจ (Function Catalog)

สารบบนี้กำหนดหมวดหน้าที่ทางธุรกิจระดับสูงขององค์กร โดยแต่ละหน้าที่สืบมาจากขีดความสามารถใน OPS-003 และมีสำนักงานเจ้าของตาม OPS-004 — รายการนี้เป็น **ตัวอย่างเชิงแนวคิด** เพื่อแสดงรูปของชั้นหน้าที่ทางธุรกิจ

| หน้าที่ทางธุรกิจ (Business Function) | สังกัดขีดความสามารถ (OPS-003) | สำนักงานเจ้าของ (OPS-004) | ประเภท |
|---|---|---|---|
| การจัดการแหล่งข้อมูล (Source Management) | Knowledge Acquisition | สำนักแหล่งข้อมูลต้นทาง | Core |
| การประมวลคลังคัมภีร์ (Corpus Processing) | Knowledge Processing | สำนักคลังคัมภีร์ | Core |
| การจัดการคลังศัพท์ (Lexicon Management) | Knowledge Processing | สำนักคลังศัพท์ | Core |
| การสร้างแบบจำลององค์ความรู้ (Knowledge Modeling) | Knowledge Processing | สำนักองค์ความรู้ | Core |
| การจัดการการค้นหา (Search Management) | Knowledge Search | สำนักองค์ความรู้ | Core |
| การจัดการความสัมพันธ์ (Relationship Management) | Knowledge Relationship | สำนักความสัมพันธ์ | Core |
| การจัดการการอ้างอิง (Citation Management) | Knowledge Validation | สำนักการอ้างอิง | Core |
| การประกันคุณภาพ (Quality Assurance) | Quality Management | สำนักคุณภาพ | Supporting |
| การสนับสนุนงานวิจัย (Research Support) | Knowledge Analytics | สำนักวิจัย | Supporting |
| การวิเคราะห์ (Analytics) | Knowledge Analytics | สำนักวิจัย | Supporting |
| การเผยแพร่ (Publishing) | Knowledge Publishing | สำนักเผยแพร่ | Core |
| การกำกับสถาปัตยกรรม (Architecture Governance) | Knowledge Governance | สำนักสถาปัตยกรรม | Governance |
| การจัดการการดำเนินงาน (Operations Management) | Knowledge Operations | สำนักปฏิบัติการ | Supporting |
| การทำงานร่วมกับมนุษย์ (Human Collaboration) | Human Collaboration | สำนักคุณภาพ | Governance |
| การช่วยเหลือด้วยผู้ช่วยอัจฉริยะ (AI Assistance) | AI Assistance | สำนักปฏิบัติการ | Supporting |

> สารบบนี้แสดงเพียง *หน้าที่ทางธุรกิจ* และที่มาของมัน **ยังไม่ลงหน้าที่ย่อย (Sub Function) หรือกระแสงาน** ซึ่งเป็นเรื่องของเอกสารระดับถัดไป

## 8. ความเป็นเจ้าของหน้าที่ (Function Ownership)

1. **หนึ่งหน้าที่ มีสำนักงานเจ้าของเพียงหนึ่งเดียว** — ไม่มีหน้าที่ทางธุรกิจใดมีเจ้าของซ้ำซ้อน
2. **หนึ่งสำนักงาน ดูแลได้หลายหน้าที่** — สำนักงานหนึ่งเป็นเจ้าของหน้าที่ทางธุรกิจได้หลายรายการ
3. **ความเป็นเจ้าของหน้าที่สอดคล้องกับความเป็นเจ้าของขีดความสามารถ** — สำนักงานเจ้าของหน้าที่ต้องเป็นเจ้าของหรือผู้สนับสนุนขีดความสามารถต้นทางตาม [OPS-004 หัวข้อ 5](PKA-OPS-004-สารบบสำนักงาน.md#5-ภาพรวมความเป็นเจ้าของขีดความสามารถ-capability-ownership-overview)

```
   สำนักวิจัย ──┬── การสนับสนุนงานวิจัย (Research Support)
                └── การวิเคราะห์ (Analytics)

   สำนักคุณภาพ ─┬── การประกันคุณภาพ (Quality Assurance)
                └── การทำงานร่วมกับมนุษย์ (Human Collaboration)

   (หนึ่งสำนักงานเป็นเจ้าของได้หลายหน้าที่ —
    แต่หน้าที่หนึ่งมีสำนักงานเจ้าของเพียงหนึ่ง)
```

## 9. การพึ่งพาระหว่างหน้าที่ (Function Dependency)

หน้าที่ทางธุรกิจมีการพึ่งพากันในเชิงองค์กร — หน้าที่หนึ่งต้องอาศัยผลของอีกหน้าที่ การพึ่งพานี้เป็น **ความสัมพันธ์เชิงหน้าที่ ไม่ใช่กระแสงาน และไม่ใช่เหตุการณ์**

```
   การประกันคุณภาพ (Quality Assurance)
        ▼  พึ่งผลของ
   การจัดการการอ้างอิง (Citation Management)
        ▼  พึ่งผลของ
   การสร้างแบบจำลององค์ความรู้ (Knowledge Modeling)
        ▼  พึ่งผลของ
   การประมวลคลังคัมภีร์ (Corpus Processing) · การจัดการคลังศัพท์ (Lexicon Management)
        ▼  พึ่งผลของ
   การจัดการแหล่งข้อมูล (Source Management)
```

**หลักการของการพึ่งพา**

1. **การพึ่งพาเป็นเชิงหน้าที่ ไม่ใช่กระแสงาน** — บอกว่าหน้าที่ใดอาศัยผลของหน้าที่ใด ไม่ใช่งานไหลอย่างไร
2. **ห้ามพึ่งพาแบบวงวน** — สายการพึ่งพาต้องไม่วนกลับเป็นวงจรปิด สอดคล้องกับการพึ่งพาขีดความสามารถใน [OPS-003 หัวข้อ 11](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#11-การพึ่งพาระหว่างขีดความสามารถ-capability-dependency)
3. **หน้าที่ด้านธรรมาภิบาลและคุณภาพกำกับขวางทุกสาย** — การประกันคุณภาพ การกำกับสถาปัตยกรรม และการทำงานร่วมกับมนุษย์ กำกับตลอดทุกการพึ่งพา ไม่ใช่อยู่ปลายสายเดียว

## 10. การจำแนกประเภทหน้าที่ (Function Classification)

| ประเภท | ความหมาย | ตัวอย่าง |
|---|---|---|
| **หน้าที่หลัก** (Core) | หน้าที่ที่เป็นแก่นของพันธกิจองค์กร | การจัดการแหล่งข้อมูล, การสร้างแบบจำลององค์ความรู้, การเผยแพร่ |
| **หน้าที่สนับสนุน** (Supporting) | หน้าที่ที่ช่วยให้หน้าที่หลักทำงานได้ | การประกันคุณภาพ, การวิเคราะห์, การจัดการการดำเนินงาน |
| **หน้าที่ธรรมาภิบาล** (Governance) | หน้าที่ที่กำกับ ควบคุม และตรวจสอบองค์กร | การกำกับสถาปัตยกรรม, การทำงานร่วมกับมนุษย์ |
| **หน้าที่นวัตกรรม** (Innovation) | หน้าที่ที่พัฒนาและปรับปรุงองค์กรในระยะยาว | การค้นคว้าและพัฒนาแนวทางใหม่ (เชิงแนวคิด) |

**หลักการ**

- หน้าที่หนึ่งจัดอยู่ในประเภทหลักเพียงประเภทเดียว เพื่อความชัดเจน
- **หน้าที่ธรรมาภิบาลต้องมีเสมอ** และไม่อาจถูกลดทอน เพราะเป็นหลักประกันความถูกต้อง
- หน้าที่นวัตกรรมต้องไม่ล่วงล้ำหลักการของเอกสารแม่ — นวัตกรรมต้องไม่สร้างหลักธรรมใหม่

## 11. วงจรชีวิตของหน้าที่ (Function Lifecycle)

```
   Proposal      ข้อเสนอ        — เสนอให้องค์กรพิจารณาว่าควรมีหน้าที่นี้
       ▼
   Approved      อนุมัติแล้ว     — ผ่านการอนุมัติให้เป็นหน้าที่ขององค์กร
       ▼
   Operational   ใช้งานจริง      — หน้าที่ทำงานได้จริงและมีสำนักงานเจ้าของดูแล
       ▼
   Deprecated    เลิกสนับสนุน    — ยังคงอยู่แต่กำลังถูกทยอยเลิก
       ▼
   Retired       ปลดระวาง        — ยุติหน้าที่ คงบันทึกไว้เพื่อการอ้างอิง
```

| สถานะ | ความหมาย |
|---|---|
| **Proposal (ข้อเสนอ)** | มีการเสนอว่าองค์กรควรมีหน้าที่ทางธุรกิจนี้ ยังไม่ผ่านการอนุมัติ |
| **Approved (อนุมัติแล้ว)** | ได้รับอนุมัติให้เป็นหน้าที่ขององค์กร และมอบหมายเจ้าของแล้ว |
| **Operational (ใช้งานจริง)** | หน้าที่ทำงานได้จริง มีสำนักงานเจ้าของดูแล |
| **Deprecated (เลิกสนับสนุน)** | กำลังถูกทยอยเลิก ยังใช้ได้แต่ไม่ส่งเสริมให้พึ่งพาต่อ |
| **Retired (ปลดระวาง)** | ยุติการใช้งานแล้ว คงบันทึกไว้เพื่อการสืบย้อน |

> วงจรชีวิตนี้สอดคล้องกับวงจรชีวิตของขีดความสามารถใน [OPS-003 หัวข้อ 8](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#8-วงจรชีวิตของขีดความสามารถ-capability-lifecycle) — การเปลี่ยนสถานะทุกครั้งต้องมีร่องรอยและการอนุมัติของมนุษย์

## 12. ธรรมาภิบาลหน้าที่ (Function Governance)

องค์กรเปลี่ยนแปลงหน้าที่ทางธุรกิจได้ห้าลักษณะ โดยทุกลักษณะต้องผ่านการอนุมัติของมนุษย์

| ลักษณะ | ความหมาย | ข้อพึงระวัง |
|---|---|---|
| **เพิ่ม** (Add) | จัดตั้งหน้าที่ทางธุรกิจใหม่ | ต้องสืบมาจากขีดความสามารถที่มีอยู่ และไม่ซ้ำหน้าที่เดิม |
| **แก้ไข** (Modify) | ปรับขอบเขตของหน้าที่เดิม | ต้องไม่เปลี่ยนแก่นจนกลายเป็นหน้าที่อื่น |
| **รวม** (Merge) | รวมหน้าที่หลายรายการเป็นหนึ่ง | ต้องรักษาความครบถ้วนของสิ่งที่เคยทำ |
| **แยก** (Split) | แยกหน้าที่หนึ่งเป็นหลายรายการ | ต้องกำหนดเจ้าของของแต่ละรายการใหม่ให้ชัด |
| **ยุบ** (Dissolve) | ยุบหน้าที่ที่ไม่จำเป็นแล้ว | ต้องตรวจว่าไม่มีหน้าที่อื่นพึ่งพาอยู่ |

**หลักการ**

1. **การเปลี่ยนหน้าที่ต้องผ่านการอนุมัติของมนุษย์** — ไม่มีหน้าที่ใดเปลี่ยนได้โดยปราศจากมนุษย์
2. **ทุกการเปลี่ยนแปลงต้องมีร่องรอย** — บันทึกในประวัติของหน้าที่และของเอกสารฉบับนี้
3. **คงหลัก "หนึ่งหน้าที่ หนึ่งเจ้าของ"** — การเปลี่ยนแปลงใด ๆ ต้องไม่ละเมิดหลักนี้
4. **สอดคล้องกับธรรมาภิบาลขีดความสามารถ** — เป็นไปตาม [OPS-003 หัวข้อ 12](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#12-ธรรมาภิบาลขีดความสามารถ-capability-governance)

## 13. คุณภาพของหน้าที่ (Function Quality)

ก่อนหน้าที่ทางธุรกิจใดเข้าสู่สถานะ *Operational* ต้องผ่านรายการตรวจคุณภาพต่อไปนี้ทุกข้อ

- ☑ **หน้าที่สืบย้อนสู่ขีดความสามารถได้** — ทุกหน้าที่ระบุขีดความสามารถต้นทางใน OPS-003 ได้
- ☑ **หน้าที่มีเจ้าของเพียงหนึ่งเดียว** — ไม่มีหน้าที่ใดมีเจ้าของซ้ำซ้อน
- ☑ **หน้าที่ไม่ทับซ้อนกับหน้าที่อื่น** — ขอบเขตของหน้าที่ชัดเจนและไม่ซ้ำ
- ☑ **หน้าที่มีประเภทชัดเจน** — ระบุประเภท (Core/Supporting/Governance/Innovation) ได้
- ☑ **การพึ่งพาไม่เป็นวงวน** — ไม่มีสายการพึ่งพาที่วนกลับ
- ☑ **ไม่มีขีดความสามารถใดไร้หน้าที่รองรับ** — ทุกขีดความสามารถใน OPS-003 มีหน้าที่ทางธุรกิจรองรับ
- ☑ **ไม่มีหน้าที่ใดไร้ขีดความสามารถต้นทาง** — ไม่มีหน้าที่กำพร้าที่ไม่สืบมาจากขีดความสามารถ

## 14. การสืบย้อนของหน้าที่ (Function Traceability)

หน้าที่ทางธุรกิจเป็นข้อต่อที่ทำให้สายเอกสารตระกูล OPS เชื่อมจากขีดความสามารถลงสู่หน่วยปฏิบัติได้ครบ

```
   Capability         (ขีดความสามารถ — OPS-003)
        ▼
   Business Function  (หน้าที่ทางธุรกิจ — เอกสารนี้, OPS-005A)
        ▼
   Workflow           (กระแสงาน — OPS-006)
        ▼
   Event              (เหตุการณ์ — OPS-007)
        ▼
   Agent              (ตัวแทนงาน — OPS-008)
        ▼
   Worker             (หน่วยปฏิบัติ — OPS-009)
```

| ชั้น | เอกสาร | เชื่อมด้วย |
|---|---|---|
| ขีดความสามารถ | OPS-003 | ชื่อขีดความสามารถ |
| **หน้าที่ทางธุรกิจ** | **OPS-005A (เอกสารนี้)** | ขีดความสามารถ → หน้าที่ทางธุรกิจ |
| กระแสงาน | OPS-006 | หน้าที่ทางธุรกิจ → กระแสงาน |
| เหตุการณ์ | OPS-007 | กระแสงาน → เหตุการณ์ |
| ตัวแทนงาน | OPS-008 | เหตุการณ์ → ตัวแทนงาน |
| หน่วยปฏิบัติ | OPS-009 | ตัวแทนงาน → หน่วยปฏิบัติ |

> **ผลของการสืบย้อน:** หน้าที่ทางธุรกิจเติมข้อต่อที่เคยขาดระหว่างขีดความสามารถกับกระแสงาน ทำให้สายการสืบย้อนสมบูรณ์ตั้งแต่ *"องค์กรทำอะไรได้"* จนถึง *"หน่วยปฏิบัติใดลงมือทำ"*

## 15. หลักการออกแบบหน้าที่ (Function Design Principles)

หน้าที่ทางธุรกิจทุกรายการต้องเป็นไปตามหลักการออกแบบอย่างน้อยสิบห้าประการต่อไปนี้

1. **ถาวร (Durable)** — หน้าที่ต้องคงทนข้ามการเปลี่ยนแปลงของผู้ทำและวิธีทำ
2. **ไม่ผูกกับบุคคล (Person-Independent)** — หน้าที่คงอยู่แม้ผู้ปฏิบัติเปลี่ยน
3. **ไม่ผูกกับผู้ช่วยอัจฉริยะ (AI-Independent)** — หน้าที่ไม่ขึ้นกับผู้ช่วยอัจฉริยะรุ่นใด
4. **ไม่ผูกกับเทคโนโลยี** — หน้าที่ไม่นิยามด้วยเครื่องมือหรือวิธีเชิงเทคนิคใด ๆ
5. **ไม่ผูกกับวิธีพัฒนา** — หน้าที่ไม่ขึ้นกับแนวทางการพัฒนาแบบใดแบบหนึ่ง
6. **ไม่ผูกกับกระแสงาน (Workflow-Independent)** — หน้าที่เป็นอิสระจากลำดับงานที่ใช้ทำ
7. **มีเจ้าของเดียว (Single Ownership)** — หน้าที่มีสำนักงานเจ้าของเพียงหนึ่งเดียว
8. **ไม่ทับซ้อน (Non-overlapping)** — ขอบเขตของหน้าที่ไม่ซ้ำกับหน้าที่อื่น
9. **ไม่ซ้ำ (Unique)** — ไม่มีหน้าที่สองรายการที่เป็นสิ่งเดียวกัน
10. **สืบย้อนสู่ขีดความสามารถได้ (Capability-Traceable)** — ทุกหน้าที่สืบมาจากขีดความสามารถใน OPS-003
11. **รองรับกระแสงานได้หลายแบบ (Workflow-Agnostic)** — หน้าที่หนึ่งรองรับกระแสงานได้หลายรูปแบบ
12. **วัดได้ (Measurable)** — ประเมินได้ว่าหน้าที่ทำงานได้ดีเพียงใด
13. **ตรวจสอบได้ (Auditable)** — การทำงานของหน้าที่สืบย้อนและตรวจสอบได้
14. **กำกับดูแลได้ (Governable)** — หน้าที่อยู่ภายใต้ธรรมาภิบาลและการอนุมัติของมนุษย์
15. **ขยายได้ (Scalable)** — รองรับงานที่เพิ่มขึ้นได้โดยไม่เปลี่ยนนิยาม
16. **มีขอบเขตชัด (Well-bounded)** — ระบุได้ชัดว่าสิ่งใดอยู่ในและสิ่งใดอยู่นอกหน้าที่
17. **สอดคล้องกับเอกสารแม่ (Aligned)** — ไม่ขัดกับหลักการใน PKA-STD-001
18. **เคารพความถูกต้องของพระธรรม (Dhamma-Respecting)** — ไม่มีหน้าที่ใดที่นำไปสู่การบิดเบือนพระธรรม

## 16. ความสัมพันธ์กับเอกสารอ้างอิง

เอกสารฉบับนี้แทรกเป็นชั้น **สถาปัตยกรรมทางธุรกิจ** ระหว่างชั้นขีดความสามารถกับชั้นการปฏิบัติ

- **PKA-OPS-003** กำหนดขีดความสามารถ — *องค์กรทำอะไรได้* และระบุว่าหน้าที่ทางธุรกิจเป็นชั้นหนึ่งในลำดับชั้น โดยรายละเอียดอยู่ในเอกสารฉบับนี้
- **PKA-OPS-004** จับคู่ขีดความสามารถกับสำนักงานเจ้าของ — *สำนักงานใดเป็นเจ้าของ*
- **PKA-OPS-005** มอบหมายขีดความสามารถสู่เจ้าของ ผู้สนับสนุน และการอนุมัติ — *มอบหมายอย่างไร*
- **PKA-OPS-005A** (ฉบับนี้) นิยามหน้าที่ถาวรขององค์กร — *องค์กรมีหน้าที่ถาวรอะไร*
- **PKA-OPS-006** จะกำหนดกระแสงานที่ทำหน้าที่เหล่านี้ — *หน้าที่ถูกทำผ่านลำดับงานใด*

เอกสารฉบับนี้อ้างอิงและต้องสอดคล้องกับ

- **[PKA-STD-001 — ธรรมนูญโครงการ](../standards/PKA-STD-001-ธรรมนูญโครงการ.md)** (เอกสารแม่)
- **[PKA-OPS-003 — แบบจำลองขีดความสามารถขององค์กร](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md)**
- **[PKA-OPS-004 — สารบบสำนักงาน](PKA-OPS-004-สารบบสำนักงาน.md)**
- **[PKA-OPS-005 — เมทริกซ์การมอบหมายขีดความสามารถ](PKA-OPS-005-เมทริกซ์การมอบหมายขีดความสามารถ.md)**

> **ลำดับชั้นยังคงเดิม:** เมื่อใดที่เนื้อหาของ OPS-005A ขัดกับเอกสารระดับสูงกว่า ให้ยึดเอกสารระดับสูงกว่าเสมอ

## 17. Roadmap

ลำดับการพัฒนาเอกสารในตระกูลปฏิบัติการ (หลังแทรกชั้นสถาปัตยกรรมทางธุรกิจ)

```
   PKA-OPS-001  : ธรรมนูญสำนักงานดิจิทัล              ✔ จัดทำแล้ว
        ▼
   PKA-OPS-002  : โครงสร้างองค์กรและสายการบังคับบัญชา    ✔ จัดทำแล้ว
        ▼
   PKA-OPS-003  : แบบจำลองขีดความสามารถขององค์กร        ✔ จัดทำแล้ว
        ▼
   PKA-OPS-004  : สารบบสำนักงาน                        ✔ จัดทำแล้ว
        ▼
   PKA-OPS-005  : เมทริกซ์การมอบหมายขีดความสามารถ      ✔ จัดทำแล้ว
        ▼
   PKA-OPS-005A : สารบบหน้าที่ทางธุรกิจ                ◀ เอกสารฉบับนี้
        ▼
   PKA-OPS-006  : สถาปัตยกรรมกระแสงาน                  ○ ขั้นถัดไป
        ▼
   PKA-OPS-007  : สถาปัตยกรรมเหตุการณ์                  ○ อนาคต
        ▼
   PKA-OPS-008  : สถาปัตยกรรมตัวแทนงาน                  ○ อนาคต
        ▼
   PKA-OPS-009  : สถาปัตยกรรมหน่วยปฏิบัติ                ○ อนาคต
```

| รหัส | ชื่อ | จุดมุ่งหมาย |
|---|---|---|
| **OPS-005A** | Business Function Catalog (สารบบหน้าที่ทางธุรกิจ) | นิยามหน้าที่ถาวรขององค์กร เชื่อมขีดความสามารถกับกระแสงาน (ฉบับนี้) |
| **OPS-006** | Workflow Architecture (สถาปัตยกรรมกระแสงาน) | กำหนดกระแสงานที่ทำหน้าที่ทางธุรกิจ |
| **OPS-007** | Event Architecture (สถาปัตยกรรมเหตุการณ์) | กำหนดเหตุการณ์ที่ใช้ประสานงานระหว่างสำนักงาน |
| **OPS-008** | Agent Architecture (สถาปัตยกรรมตัวแทนงาน) | กำหนดตัวแทนงานจริงและภารกิจภายในสำนักงาน |
| **OPS-009** | Worker Architecture (สถาปัตยกรรมหน่วยปฏิบัติ) | กำหนดหน่วยปฏิบัติจริงและงานหนึ่งหน่วย |
