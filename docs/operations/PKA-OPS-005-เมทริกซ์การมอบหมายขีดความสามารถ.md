## 1. วัตถุประสงค์ (Purpose)

เอกสารฉบับนี้กำหนด **เมทริกซ์การมอบหมายขีดความสามารถ** ซึ่งระบุว่าขีดความสามารถแต่ละรายการขององค์กร — ที่นิยามไว้ใน [PKA-OPS-003](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md) และจับคู่กับสำนักงานเจ้าของใน [PKA-OPS-004](PKA-OPS-004-สารบบสำนักงาน.md) — จะถูกมอบหมายความรับผิดชอบ การปฏิบัติในอนาคต และการอนุมัติอย่างไร

เอกสารนี้มีไว้เพื่อตอบคำถามต่อขีดความสามารถแต่ละรายการว่า

1. **ใครเป็นเจ้าของ** — สำนักงานเจ้าของเพียงหนึ่งเดียว
2. **ใครสนับสนุน** — สำนักงานผู้สนับสนุน
3. **อนาคตตัวแทนงานประเภทใดจะรับผิดชอบ** — ประเภทตัวแทนงาน (Agent Type) เชิงแนวคิด
4. **อนาคตหน่วยปฏิบัติประเภทใดจะลงมือทำ** — ประเภทหน่วยปฏิบัติ (Worker Type) เชิงแนวคิด
5. **มนุษย์ระดับใดเป็นผู้อนุมัติ** — ระดับอำนาจการอนุมัติ

> เอกสารฉบับนี้เป็น **สะพานเชื่อมโลกสองใบ** — โลกของ *สถาปัตยกรรมองค์กร (Organization Architecture)* ที่บอกว่าองค์กรทำอะไรได้และใครเป็นเจ้าของ กับโลกของ *สถาปัตยกรรมการปฏิบัติ (Execution Architecture)* ที่บอกว่างานจะถูกทำอย่างไร โดยทุกอย่างยังเป็น **แนวคิด** — ยังไม่ระบุชื่อตัวแทนงานจริง และยังไม่สร้างหน่วยปฏิบัติจริง

## 2. ขอบเขต (Scope)

**อยู่ในขอบเขต**

- การมอบหมายขีดความสามารถแต่ละรายการให้สำนักงานเจ้าของและสำนักงานผู้สนับสนุน
- การเชื่อมขีดความสามารถสู่ประเภทตัวแทนงานและประเภทหน่วยปฏิบัติในอนาคต (เชิงแนวคิด)
- ระดับอำนาจการอนุมัติของขีดความสามารถแต่ละประเภท
- การพึ่งพาระหว่างสำนักงานในเชิงองค์กร วงจรชีวิตของการมอบหมาย ธรรมาภิบาล และคุณภาพของการมอบหมาย
- การสืบย้อนเชื่อมต่อตลอดสายเอกสาร OPS-003 ถึง OPS-009

**อยู่นอกขอบเขต**

- หน้าที่ทางธุรกิจโดยละเอียด (Business Function) — อยู่ใน **PKA-OPS-005A**
- กระแสงาน (Workflow) — อยู่ใน **PKA-OPS-006**
- เหตุการณ์ (Event) — อยู่ใน **PKA-OPS-007**
- ตัวแทนงานจริงและชื่อจริง — อยู่ใน **PKA-OPS-008**
- หน่วยปฏิบัติจริง — อยู่ใน **PKA-OPS-009**
- คิวงาน กระบวนการ การทำงานอัตโนมัติ และรายละเอียดเชิงปฏิบัติทั้งหมด
- เครื่องมือ เทคโนโลยี วิธีพัฒนา หรือรายละเอียดเชิงเทคนิคใด ๆ

> เอกสารฉบับนี้หยุดที่ **"การมอบหมาย"** — บอกว่าใครรับผิดชอบและใครอนุมัติ แต่ **ไม่บอกว่างานไหลอย่างไร** ซึ่งเป็นเรื่องของ OPS-006 เป็นต้นไป

## 3. การมอบหมายคืออะไร (What is Assignment)

**การมอบหมาย (Assignment)** คือการผูกขีดความสามารถหนึ่งเข้ากับ *ผู้รับผิดชอบ* *ผู้ปฏิบัติในอนาคต* และ *ผู้อนุมัติ* อย่างชัดเจน เพื่อให้องค์กรรู้ว่าความสามารถแต่ละอย่างมีใครดูแลตลอดเส้นทางจากการเป็นเจ้าของไปจนถึงการอนุมัติ

การมอบหมายในเอกสารนี้ครอบคลุมเส้นทางหกขั้น

```
   Capability            (ขีดความสามารถ — OPS-003)
        ▼
   Owner Office          (สำนักงานเจ้าของ — OPS-004)
        ▼
   Supporting Office(s)  (สำนักงานผู้สนับสนุน — OPS-004)
        ▼
   Future Agent Type     (ประเภทตัวแทนงานในอนาคต — จะนิยามจริงใน OPS-008)
        ▼
   Future Worker Type    (ประเภทหน่วยปฏิบัติในอนาคต — จะนิยามจริงใน OPS-009)
        ▼
   Human Approval Authority  (อำนาจการอนุมัติของมนุษย์)
```

**ความแตกต่างจากแนวคิดข้างเคียง**

- **การมอบหมายไม่ใช่ความเป็นเจ้าของเพียงอย่างเดียว** — OPS-004 ตอบว่า *ใครเป็นเจ้าของ* ส่วน OPS-005 ขยายไปถึง *ใครสนับสนุน ใครปฏิบัติในอนาคต และใครอนุมัติ*
- **การมอบหมายไม่ใช่กระแสงาน** — การมอบหมายบอกว่า *ใครรับผิดชอบ* ไม่ใช่ *งานไหลอย่างไร* (ซึ่งเป็นของ OPS-006)
- **การมอบหมายไม่ใช่เมทริกซ์ RACI** — แม้มีองค์ประกอบคล้ายกัน แต่หัวใจของเอกสารนี้คือการ **เชื่อมขีดความสามารถสู่การปฏิบัติในอนาคต** ไม่ใช่การจัดบทบาทผู้รับผิดชอบในงานเฉพาะ

## 4. หลักการของการมอบหมาย (Assignment Principle)

หลักการสูงสุดของการมอบหมายคือ **"ขีดความสามารถผูกกับองค์กร ไม่ผูกกับผู้ทำหรือวิธีทำ"**

1. **ไม่ผูกกับบุคคล** — การมอบหมายผูกขีดความสามารถกับ *สำนักงาน* ไม่ใช่กับบุคคลใดบุคคลหนึ่ง
2. **ไม่ผูกกับผู้ช่วยอัจฉริยะ (AI)** — การมอบหมายอ้างถึง *ประเภท* ตัวแทนงาน ไม่ใช่ผู้ช่วยอัจฉริยะรุ่นใดรุ่นหนึ่ง
3. **ไม่ผูกกับกระแสงาน (Workflow)** — การมอบหมายดำรงอยู่ก่อนและเป็นอิสระจากกระแสงานที่จะออกแบบใน OPS-006
4. **ผูกกับองค์กร** — ความรับผิดชอบต่อขีดความสามารถเป็นขององค์กรผ่านสำนักงานเจ้าของ ซึ่งคงทนข้ามการเปลี่ยนแปลง

ผลของหลักการนี้คือความยืดหยุ่นที่มั่นคง

```
   องค์กรเปลี่ยน  ตัวแทนงาน (Agent)   ได้
   องค์กรเปลี่ยน  หน่วยปฏิบัติ (Worker) ได้
   องค์กรเปลี่ยน  กระแสงาน (Workflow)  ได้
   องค์กรเปลี่ยน  ผู้ช่วยอัจฉริยะ (AI)  ได้
   ─────────────────────────────────────────
   แต่  ขีดความสามารถ (Capability)  ไม่เปลี่ยน
```

> หลักการนี้ต่อยอดโดยตรงจาก [OPS-003 หัวข้อ 7](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#7-ขีดความสามารถไม่ผูกกับสิ่งใด-capability-independence) และ "องค์กรถาวร กำลังพลทดแทนได้" ใน [OPS-002 หัวข้อ 21](PKA-OPS-002-โครงสร้างองค์กรและสายการบังคับบัญชา.md#21-องค์กรถาวร-กำลังพลทดแทนได้-stable-organization--replaceable-workforce) — การมอบหมายจึงเปลี่ยนผู้ปฏิบัติได้โดยขีดความสามารถยังคงเดิม

## 5. เมทริกซ์การมอบหมายขีดความสามารถ (Capability Assignment Matrix)

เมทริกซ์กลางนี้แสดงการมอบหมายขีดความสามารถระดับบนทั้งสิบสองรายการจาก [OPS-003 หัวข้อ 15](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#15-แผนที่ขีดความสามารถ-capability-map) เพื่อความอ่านง่ายจึงแบ่งเป็นสองตารางที่เชื่อมกันด้วยชื่อขีดความสามารถ

| AI Assistance | สำนักปฏิบัติการ | — | ผู้บริหารงานองค์กร (Executive) + มนุษย์เมื่อผลเข้าสู่องค์ความรู้ | §15 (สนับสนุน) | §6.11 |
| Quality Management | สำนักคุณภาพ | สำนักการอ้างอิง, สำนักปฏิบัติการ | สำนักงาน (ยกระดับสู่ผู้บริหารเมื่อเป็นเกณฑ์ทั้งองค์กร) | §15 (สนับสนุน) | §6.7 |
| Knowledge Governance | สำนักคุณภาพ | สำนักแหล่งข้อมูลต้นทาง, สำนักการอ้างอิง, สำนักเผยแพร่, สำนักสถาปัตยกรรม, สำนักปฏิบัติการ | มนุษย์ / คณะกรรมการในอนาคต | §15 (ธรรมาภิบาล) | §6.7 |
| Human Collaboration | สำนักคุณภาพ | — | มนุษย์ | §15 (ธรรมาภิบาล) | §6.7 |

> สำหรับขีดความสามารถย่อยสามรายการ (Corpus Structuring → สำนักคลังคัมภีร์, Lexical Standardization → สำนักคลังศัพท์, Structural Consistency → สำนักสถาปัตยกรรม) การมอบหมายเป็นไปตามขีดความสามารถแม่ของแต่ละรายการตาม [OPS-004 หัวข้อ 5](PKA-OPS-004-สารบบสำนักงาน.md#5-ภาพรวมความเป็นเจ้าของขีดความสามารถ-capability-ownership-overview)

### 5.2 ตารางการปฏิบัติในอนาคต (Future Execution Assignment)

ตารางนี้เชื่อมขีดความสามารถสู่ **ประเภท** ตัวแทนงานและหน่วยปฏิบัติในอนาคต ทั้งหมดเป็น **แนวคิดเชิงประเภท** ยังไม่ใช่ตัวแทนงานหรือหน่วยปฏิบัติจริง

| ขีดความสามารถ | ประเภทตัวแทนงานในอนาคต (Agent Type) | ประเภทหน่วยปฏิบัติในอนาคต (Worker Type) | ระดับการปฏิบัติ |
|---|---|---|---|
| Knowledge Acquisition | ตัวแทนรับเข้า (Acquisition Agent) | หน่วยขึ้นทะเบียนแหล่งข้อมูล (Source Registration Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Processing | ตัวแทนประมวล (Processing Agent) | หน่วยจัดโครงสร้างเนื้อหา (Structuring Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Search | ตัวแทนค้นหา (Search Agent) | หน่วยจัดเตรียมการค้นหา (Search Preparation Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Validation | ตัวแทนตรวจสอบ (Validation Agent) | หน่วยตรวจการอ้างอิง (Citation Check Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Relationship | ตัวแทนเชื่อมโยง (Relationship Agent) | หน่วยสร้างความเชื่อมโยง (Linking Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Publishing | ตัวแทนเผยแพร่ (Publishing Agent) | หน่วยจัดเตรียมเผยแพร่ (Publication Preparation Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Analytics | ตัวแทนวิเคราะห์ (Analytics Agent) | หน่วยวิเคราะห์ (Analysis Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Operations | ตัวแทนประสานงาน (Operations Agent) | หน่วยติดตามสถานะ (Monitoring Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| AI Assistance | ตัวแทนผู้ช่วยอัจฉริยะ (Assistance Agent) | หน่วยช่วยเหลือ (Assist Worker) | ตัวแทนงาน + หน่วยปฏิบัติ (ใต้การกำกับ) |
| Quality Management | ตัวแทนคุณภาพ (Quality Agent) | หน่วยตรวจคุณภาพ (Quality Check Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Knowledge Governance | ตัวแทนกำกับ (Governance Agent) | หน่วยตรวจสอบ (Audit Worker) | ตัวแทนงาน + หน่วยปฏิบัติ |
| Human Collaboration | ตัวแทนความร่วมมือมนุษย์ (Collaboration Agent) | หน่วยประสานการทบทวน (Review Coordination Worker) | ตัวแทนงาน + หน่วยปฏิบัติ + มนุษย์ |

> **ย้ำ:** ชื่อในตารางนี้เป็น **ประเภทเชิงแนวคิด** เพื่อแสดงว่าขีดความสามารถจะถูกปฏิบัติโดยตัวแทนงานและหน่วยปฏิบัติประเภทใดในอนาคต ตัวแทนงานจริงจะนิยามใน **OPS-008** และหน่วยปฏิบัติจริงใน **OPS-009** เอกสารฉบับนี้ **ไม่สร้าง** ตัวแทนงานหรือหน่วยปฏิบัติใด ๆ

## 6. การพึ่งพาระหว่างสำนักงาน (Office Dependency)

นอกจากการมอบหมายต่อขีดความสามารถแล้ว สำนักงานยังมี **การพึ่งพาเชิงองค์กร (Organizational Dependency)** ต่อกัน — สำนักงานหนึ่งต้องอาศัยผลผลิตของอีกสำนักงานจึงทำงานของตนได้ การพึ่งพานี้เป็น **ความสัมพันธ์เชิงโครงสร้าง ไม่ใช่กระแสงาน**

### 6.1 ตัวอย่างสายการพึ่งพา

```
   สำนักคุณภาพ
        ▼  พึ่งผลของ
   สำนักการอ้างอิง
        ▼  พึ่งผลของ
   สำนักคลังคัมภีร์ · สำนักคลังศัพท์
        ▼  พึ่งผลของ
   สำนักแหล่งข้อมูลต้นทาง   (การรับองค์ความรู้)
```

```
   สำนักปฏิบัติการ (กำกับผู้ช่วยอัจฉริยะ)
        ▼  พึ่งผลของ
   สำนักองค์ความรู้
        ▼  พึ่งผลของ
   การค้นหา (Knowledge Search)
        ▼  อยู่ภายใต้การกำกับของ
   สำนักคุณภาพ
```

### 6.2 ตารางการพึ่งพาเชิงองค์กร

| สำนักงาน | พึ่งพาผลของสำนักงาน (ต้นทาง) |
|---|---|
| สำนักคลังคัมภีร์ | สำนักแหล่งข้อมูลต้นทาง |
| สำนักคลังศัพท์ | สำนักคลังคัมภีร์ |
| สำนักองค์ความรู้ | สำนักคลังคัมภีร์, สำนักคลังศัพท์ |
| สำนักความสัมพันธ์ | สำนักองค์ความรู้ |
| สำนักการอ้างอิง | สำนักองค์ความรู้, สำนักความสัมพันธ์ |
| สำนักคุณภาพ | สำนักการอ้างอิง |
| สำนักเผยแพร่ | สำนักคุณภาพ (หลังการอนุมัติของมนุษย์) |
| สำนักวิจัย | สำนักองค์ความรู้, สำนักความสัมพันธ์ |
| สำนักสถาปัตยกรรม | (กำกับขวางทุกสำนักงานในเชิงโครงสร้าง) |
| สำนักปฏิบัติการ | ทุกสำนักงาน (ในฐานะผู้ประสานและติดตาม) |

**หลักการของการพึ่งพา**

1. **การพึ่งพาเป็นเชิงองค์กร ไม่ใช่การเรียกโดยตรง** — สำนักงานพึ่ง *ผลผลิต* ของกันและกัน แต่ไม่เรียกกันโดยตรง การประสานจริงผ่านเหตุการณ์ตาม [OPS-002 หัวข้อ 16](PKA-OPS-002-โครงสร้างองค์กรและสายการบังคับบัญชา.md#16-การประสานงานและหลักการสื่อสาร-coordination--communication-principles)
2. **ห้ามพึ่งพาเป็นวงวน** — สายการพึ่งพาต้องไม่วนกลับเป็นวงจรปิด สอดคล้องกับการพึ่งพาขีดความสามารถใน [OPS-003 หัวข้อ 11](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#11-การพึ่งพาระหว่างขีดความสามารถ-capability-dependency)
3. **สำนักคุณภาพและสำนักสถาปัตยกรรมกำกับขวางทุกสาย** — มิได้อยู่ปลายสายเดียว แต่กำกับความถูกต้องและความสอดคล้องตลอดทุกการพึ่งพา

## 7. อำนาจการอนุมัติ (Approval Authority)

ขีดความสามารถแต่ละประเภทต้องได้รับการอนุมัติที่ระดับที่เหมาะสม องค์กรกำหนดระดับอำนาจการอนุมัติไว้สี่ระดับ

| ระดับ | ผู้อนุมัติ | ใช้กับ |
|---|---|---|
| **ระดับสำนักงาน** (Office) | สำนักงานเจ้าของ | การอนุมัติเชิงดำเนินงานภายในขีดความสามารถ |
| **ระดับผู้บริหาร** (Executive) | ผู้บริหารงานองค์กร / ผู้อำนวยการสูงสุด | เรื่องที่กระทบหลายสำนักงานหรือทั้งองค์กร |
| **ระดับมนุษย์** (Human) | มนุษย์ผู้ทบทวน | การอนุมัติองค์ความรู้และเรื่องที่กระทบความถูกต้องของพระธรรม |
| **ระดับคณะกรรมการในอนาคต** (Future Committee) | คณะกรรมการที่จะจัดตั้ง | เรื่องที่กระทบหลักธรรมหรือโครงสร้างองค์กรอย่างมีนัยสำคัญ (เชิงแนวคิด) |

**หลักการของอำนาจการอนุมัติ**

1. **การอนุมัติองค์ความรู้เป็นของมนุษย์เสมอ** — ไม่ว่าระดับใด ขีดความสามารถที่นำสู่การเผยแพร่องค์ความรู้ต้องผ่านการอนุมัติของมนุษย์ ตามเอกสารแม่และ [OPS-001 หัวข้อ 19](PKA-OPS-001-ธรรมนูญสำนักงานดิจิทัล.md#19-human-in-the-loop--มนุษย์ในวงจรการตัดสินใจ)
2. **ผู้ช่วยอัจฉริยะไม่มีระดับอนุมัติ** — ผู้ช่วยอัจฉริยะไม่ปรากฏในตารางนี้ เพราะไม่มีอำนาจอนุมัติองค์ความรู้ในระดับใดเลย
3. **ระดับการอนุมัติยกระดับขึ้นได้ตามความเสี่ยง** — ยิ่งเรื่องกระทบความถูกต้องของพระธรรมมาก ระดับการอนุมัติยิ่งสูงขึ้น
4. **คณะกรรมการในอนาคตเป็นเพียงแนวคิด** — ระดับนี้สงวนไว้เผื่อองค์กรเติบโต โดยยังไม่กำหนดรายละเอียดเชิงปฏิบัติในฉบับนี้

## 8. วงจรชีวิตของการมอบหมาย (Assignment Lifecycle)

การมอบหมายแต่ละรายการมีวงจรชีวิตของตนเอง ซึ่งต่างจากวงจรชีวิตของขีดความสามารถใน OPS-003

```
   Draft        ร่างการมอบหมาย     — เสนอการมอบหมายขีดความสามารถ ยังไม่ยืนยัน
       ▼
   Assigned     มอบหมายแล้ว        — ระบุเจ้าของ ผู้สนับสนุน และระดับการอนุมัติครบ
       ▼
   Verified     ตรวจสอบแล้ว         — ผ่านการตรวจว่าสืบย้อนได้และไม่มีช่องว่าง
       ▼
   Operational  ใช้งานจริง          — การมอบหมายมีผลและพร้อมเชื่อมสู่การปฏิบัติ
       ▼
   Retired      ปลดระวาง            — ยกเลิกการมอบหมาย คงบันทึกไว้เพื่ออ้างอิง
```

| สถานะ | ความหมาย |
|---|---|
| **Draft (ร่าง)** | เสนอการมอบหมายขีดความสามารถ ยังไม่ผ่านการยืนยัน |
| **Assigned (มอบหมายแล้ว)** | ระบุสำนักงานเจ้าของ ผู้สนับสนุน และระดับการอนุมัติครบถ้วน |
| **Verified (ตรวจสอบแล้ว)** | ผ่านการตรวจคุณภาพการมอบหมาย — สืบย้อนได้และไม่มีช่องว่าง |
| **Operational (ใช้งานจริง)** | การมอบหมายมีผล พร้อมเชื่อมสู่ประเภทตัวแทนงานและหน่วยปฏิบัติในอนาคต |
| **Retired (ปลดระวาง)** | ยกเลิกการมอบหมาย เมื่อขีดความสามารถถูกปลดระวางหรือมอบหมายใหม่ |

> วงจรนี้สอดคล้องกับวงจรชีวิตของขีดความสามารถใน [OPS-003 หัวข้อ 8](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#8-วงจรชีวิตของขีดความสามารถ-capability-lifecycle) แต่เป็นคนละชั้น — ชั้นนี้ว่าด้วย *การมอบหมาย* ของขีดความสามารถที่มีอยู่ ไม่ใช่การมีอยู่ของขีดความสามารถเอง

## 9. ธรรมาภิบาลการมอบหมาย (Assignment Governance)

การเปลี่ยนแปลงการมอบหมายอยู่ภายใต้ธรรมาภิบาลที่ชัดเจน

| การกระทำ | ผู้มีสิทธิ์ | เงื่อนไข |
|---|---|---|
| **เพิ่มการมอบหมาย** | ผู้บริหารงานองค์กรเสนอ + มนุษย์อนุมัติ | ขีดความสามารถต้องมีอยู่ใน OPS-003 และมีเจ้าของใน OPS-004 |
| **แก้การมอบหมาย** | สำนักงานเจ้าของเสนอ + มนุษย์อนุมัติ | ต้องไม่ทำให้เกิดขีดความสามารถไร้เจ้าภาพ |
| **ยกเลิกการมอบหมาย** | ผู้บริหารงานองค์กรเสนอ + มนุษย์อนุมัติ | ต้องตรวจว่าไม่มีสำนักงานอื่นพึ่งพาอยู่ |

**หลักการของธรรมาภิบาลการมอบหมาย**

1. **การเปลี่ยนการมอบหมายต้องผ่านการอนุมัติของมนุษย์** — ไม่มีการมอบหมายใดเปลี่ยนได้โดยปราศจากมนุษย์
2. **ทุกการเปลี่ยนแปลงต้องมีร่องรอย** — บันทึกในประวัติของการมอบหมายและของเอกสารฉบับนี้
3. **การมอบหมายต้องคงหลัก "หนึ่งขีดความสามารถ หนึ่งเจ้าของ"** — การเปลี่ยนแปลงใด ๆ ต้องไม่ละเมิดหลักนี้
4. **สอดคล้องกับธรรมาภิบาลขีดความสามารถ** — เป็นไปตาม [OPS-003 หัวข้อ 12](PKA-OPS-003-แบบจำลองขีดความสามารถขององค์กร.md#12-ธรรมาภิบาลขีดความสามารถ-capability-governance)

## 10. คุณภาพของการมอบหมาย (Assignment Quality)

ก่อนการมอบหมายใดจะเข้าสู่สถานะ *Verified* ต้องผ่านรายการตรวจคุณภาพต่อไปนี้ทุกข้อ

- ☑ **ขีดความสามารถมีเจ้าของเพียงหนึ่งเดียว** — ไม่มีขีดความสามารถใดมีเจ้าของซ้ำซ้อน
- ☑ **สำนักงานผู้สนับสนุนมีเหตุผลรองรับ** — การกำหนดผู้สนับสนุนต้องอธิบายได้ว่าสนับสนุนอย่างไร
- ☑ **ระดับการอนุมัติชัดเจน** — ทุกขีดความสามารถระบุระดับการอนุมัติที่แน่นอน
- ☑ **สืบย้อนกลับได้** — ทุกการมอบหมายสืบย้อนสู่ OPS-003 และ OPS-004 ได้
- ☑ **ไม่มีขีดความสามารถกำพร้า** — ไม่มีขีดความสามารถใดใน OPS-003 ที่ไม่มีการมอบหมาย
- ☑ **ไม่มีสำนักงานที่ไร้ขีดความสามารถ** — ทุกสำนักงานใน OPS-004 เป็นเจ้าของอย่างน้อยหนึ่งขีดความสามารถหรือขีดความสามารถย่อย

> รายการตรวจคุณภาพนี้คือเกณฑ์ที่รับประกันว่าเมทริกซ์การมอบหมาย **ครบถ้วน ไม่มีช่องว่าง และตรวจสอบได้** ก่อนนำไปใช้เชื่อมสู่ชั้นการปฏิบัติ

## 11. การสืบย้อนทั้งสายเอกสาร (Traceability)

เอกสารฉบับนี้เป็นจุดเชื่อมที่ทำให้สายเอกสารตระกูล OPS ต่อกันครบ ตั้งแต่ขีดความสามารถจนถึงหน่วยปฏิบัติ

```
   OPS-003  Capability            องค์กรทำอะไรได้
        │   ── ขีดความสามารถ ──
        ▼
   OPS-004  Office Catalog        สำนักงานใดเป็นเจ้าของ
        │   ── เจ้าของ ──
        ▼
   OPS-005  Assignment Matrix     มอบหมายให้ใคร อนุมัติที่ระดับใด   ◀ เอกสารฉบับนี้
        │   ── การมอบหมาย ──
        ▼
   OPS-005A Business Function     องค์กรมีหน้าที่ถาวรอะไร
        │   ── หน้าที่ทางธุรกิจ ──
        ▼
   OPS-006  Workflow Architecture งานไหลอย่างไร
        │   ── กระแสงาน ──
        ▼
   OPS-007  Event Architecture    ประสานด้วยเหตุการณ์ใด
        │   ── เหตุการณ์ ──
        ▼
   OPS-008  Agent Architecture    ตัวแทนงานใดรับผิดชอบ
        │   ── ตัวแทนงาน ──
        ▼
   OPS-009  Worker Architecture   หน่วยปฏิบัติใดลงมือทำ
```

| เอกสาร | นิยามอะไร | เชื่อมด้วย |
|---|---|---|
| OPS-003 | ขีดความสามารถขององค์กร | ชื่อขีดความสามารถ |
| OPS-004 | สำนักงานเจ้าของและผู้สนับสนุน | ขีดความสามารถ → สำนักงาน |
| **OPS-005** | การมอบหมาย การอนุมัติ และการเชื่อมสู่การปฏิบัติในอนาคต | ขีดความสามารถ → เจ้าของ → ผู้สนับสนุน → ประเภทตัวแทนงาน/หน่วยปฏิบัติ → การอนุมัติ |
| OPS-005A | หน้าที่ทางธุรกิจ (หน้าที่ถาวรขององค์กร) | ขีดความสามารถ → หน้าที่ทางธุรกิจ |
| OPS-006 | กระแสงานที่ทำหน้าที่ทางธุรกิจ | หน้าที่ทางธุรกิจ → กระแสงาน |
| OPS-007 | เหตุการณ์ที่ใช้ประสานงาน | กระแสงาน → เหตุการณ์ |
| OPS-008 | ตัวแทนงานจริง | ประเภทตัวแทนงาน → ตัวแทนงานจริง |
| OPS-009 | หน่วยปฏิบัติจริง | ประเภทหน่วยปฏิบัติ → หน่วยปฏิบัติจริง |

> **ผลของการสืบย้อน:** OPS-005 คือข้อต่อที่ทำให้สายจาก *"องค์กรทำอะไรได้"* (OPS-003) เชื่อมต่อถึง *"หน่วยปฏิบัติใดลงมือทำ"* (OPS-009) ได้อย่างไม่ขาดตอน โดยฉบับนี้รับผิดชอบเฉพาะข้อต่อ **การมอบหมาย** เท่านั้น

## 12. Roadmap

ลำดับการพัฒนาเอกสารในตระกูลปฏิบัติการ

```
   PKA-OPS-001 : ธรรมนูญสำนักงานดิจิทัล              ✔ จัดทำแล้ว
        ▼
   PKA-OPS-002 : โครงสร้างองค์กรและสายการบังคับบัญชา    ✔ จัดทำแล้ว
        ▼
   PKA-OPS-003 : แบบจำลองขีดความสามารถขององค์กร        ✔ จัดทำแล้ว
        ▼
   PKA-OPS-004 : สารบบสำนักงาน                        ✔ จัดทำแล้ว
        ▼
   PKA-OPS-005 : เมทริกซ์การมอบหมายขีดความสามารถ      ◀ เอกสารฉบับนี้
        ▼
   PKA-OPS-005A: สารบบหน้าที่ทางธุรกิจ                ○ ขั้นถัดไป
        ▼
   PKA-OPS-006 : สถาปัตยกรรมกระแสงาน                  ○ อนาคต
        ▼
   PKA-OPS-007 : สถาปัตยกรรมเหตุการณ์                  ○ อนาคต
        ▼
   PKA-OPS-008 : สถาปัตยกรรมตัวแทนงาน                  ○ อนาคต
        ▼
   PKA-OPS-009 : สถาปัตยกรรมหน่วยปฏิบัติ                ○ อนาคต
```

| รหัส | ชื่อ | จุดมุ่งหมาย |
|---|---|---|
| **OPS-005** | Capability Assignment Matrix (เมทริกซ์การมอบหมายขีดความสามารถ) | สะพานเชื่อมขีดความสามารถสู่การมอบหมายและการอนุมัติ (ฉบับนี้) |
| **OPS-005A** | Business Function Catalog (สารบบหน้าที่ทางธุรกิจ) | นิยามหน้าที่ถาวรขององค์กร เชื่อมขีดความสามารถกับกระแสงาน |
| **OPS-006** | Workflow Architecture (สถาปัตยกรรมกระแสงาน) | กำหนดกระแสงานที่ทำหน้าที่ทางธุรกิจ |
| **OPS-007** | Event Architecture (สถาปัตยกรรมเหตุการณ์) | กำหนดเหตุการณ์ที่ใช้ประสานงานระหว่างสำนักงาน |
| **OPS-008** | Agent Architecture (สถาปัตยกรรมตัวแทนงาน) | กำหนดตัวแทนงานจริงและภารกิจภายในสำนักงาน |
| **OPS-009** | Worker Architecture (สถาปัตยกรรมหน่วยปฏิบัติ) | กำหนดหน่วยปฏิบัติจริงและงานหนึ่งหน่วย |
