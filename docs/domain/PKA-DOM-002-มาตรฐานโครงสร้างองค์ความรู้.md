## 1. วัตถุประสงค์

เอกสารฉบับนี้กำหนด **โครงสร้างมาตรฐานขององค์ความรู้หนึ่งรายการ** เพื่อให้องค์ความรู้ทุกประเภทในโครงการมีรูปแบบเดียวกัน ตรวจสอบย้อนกลับได้ และมีคุณภาพที่วัดได้

เอกสารฉบับนี้เป็น **แม่แบบ (Schema เชิงแนวคิด)** ที่จะถูกนำไปใช้กับองค์ความรู้ทุกชนิดที่กำหนดไว้ใน [PKA-DOM-001](PKA-DOM-001-แบบจำลองโดเมนองค์ความรู้พระธรรม.md) โดย:

- **ยังไม่สร้างองค์ความรู้จริง** — เป็นเพียงการกำหนดรูปทรงของภาชนะ
- **ไม่ผูกกับองค์ธรรมใด ๆ** — แม่แบบต้องใช้ได้กับองค์ความรู้ทุกประเภทอย่างเสมอภาค
- **ไม่ใช่การออกแบบระบบจัดเก็บ** — ทุกองค์ประกอบเป็นแนวคิด ไม่ใช่ชื่อฟิลด์ของฐานข้อมูล

## 2. ขอบเขต

### 2.1 อยู่ในขอบเขต

- โครงสร้างมาตรฐานขององค์ความรู้หนึ่งรายการในเชิงแนวคิด
- การจำแนกองค์ประกอบเป็น ต้องมี / ควรมี / เพิ่มเติม
- หลักการกำหนดรหัส ชื่อ คำอธิบาย การอ้างอิง ความสัมพันธ์ สถานะ เวอร์ชัน และข้อมูลกำกับ
- วงจรชีวิต ระดับความสมบูรณ์ และการตรวจสอบคุณภาพขององค์ความรู้

### 2.2 อยู่นอกขอบเขต

- การออกแบบฐานข้อมูล โครงสร้างการจัดเก็บ หรือชื่อฟิลด์จริง
- การออกแบบ JSON, SQL, API หรือรูปแบบข้อมูลเชิงเทคนิคใด ๆ
- เครื่องมือ เทคโนโลยี และรายละเอียดการ Implement
- เนื้อหาพระธรรม องค์ธรรม และหมวดธรรม
- รายการจริง (instance) ขององค์ความรู้

---

## 3. หลักการ

1. **แม่แบบเดียวสำหรับทุกประเภท** — องค์ความรู้ทุกชนิดใช้โครงสร้างมาตรฐานเดียวกัน
2. **เป็นกลางต่อเนื้อหา** — แม่แบบต้องไม่ผูกกับองค์ธรรมหรือหมวดธรรมใด
3. **ตรวจสอบย้อนกลับได้เสมอ** — ทุกองค์ความรู้ต้องสืบกลับไปยังแหล่งข้อมูลหลักได้
4. **แยกเนื้อหาต้นทางออกจากคำอธิบาย** — ต้องแยกได้ว่าส่วนใดเป็นเนื้อหาต้นทาง ส่วนใดเป็นคำอธิบายของโครงการ
5. **คุณภาพต้องวัดได้** — ความสมบูรณ์และคุณภาพขององค์ความรู้ต้องประเมินได้ด้วยเกณฑ์ที่ชัดเจน
6. **เรียบง่ายและขยายได้** — แม่แบบต้องเรียบง่ายที่สุดเท่าที่ครอบคลุมได้ และเปิดให้เพิ่มองค์ประกอบในอนาคตอย่างมีระเบียบ
7. **แนวคิดมาก่อนการจัดเก็บ** — เอกสารนี้กำหนด *ความหมาย* ขององค์ประกอบ ไม่ใช่ *วิธีจัดเก็บ*

## 4. นิยามของ "องค์ความรู้"

> **องค์ความรู้ (Knowledge Unit)** หมายถึง หน่วยความรู้หนึ่งรายการที่ระบบจัดการเป็นเอกเทศได้ มีรหัสประจำตัวเฉพาะ มีโครงสร้างตามแม่แบบนี้ และสามารถสืบย้อนกลับไปยังแหล่งข้อมูลหลักได้

คุณสมบัติพื้นฐานขององค์ความรู้หนึ่งรายการ:

- เป็นหน่วยที่ **ระบุตัวตนได้เฉพาะ** ด้วยรหัสประจำตัว
- มี **ระดับความน่าเชื่อถือ** กำกับตามลำดับชั้นแหล่งข้อมูลในเอกสารแม่
- มี **สถานะ** ที่บอกตำแหน่งในวงจรชีวิต
- เชื่อมโยงกับองค์ความรู้อื่นได้ผ่าน **ความสัมพันธ์**
- นิยามนี้เป็นเชิงโครงสร้าง ไม่ใช่นิยามทางพระธรรม

## 5. ส่วนประกอบมาตรฐานขององค์ความรู้ (แม่แบบ)

ตารางต่อไปนี้คือ **แม่แบบองค์ความรู้** แสดงองค์ประกอบทั้งหมดในเชิงแนวคิด พร้อมระดับความจำเป็น

> หมายเหตุ: คอลัมน์ "องค์ประกอบ" เป็น *ชื่อเชิงแนวคิด* ไม่ใช่ชื่อฟิลด์ของฐานข้อมูล

| องค์ประกอบ (แนวคิด) | ระดับความจำเป็น | คำอธิบาย |
|---|---|---|
| รหัส | ต้องมี | ตัวระบุประจำตัวเฉพาะขององค์ความรู้ |
| ชื่อ | ต้องมี | ชื่อหลักขององค์ความรู้ (ภาษาไทย) |
| ประเภท | ต้องมี | ชนิดขององค์ความรู้ตาม PKA-DOM-001 |
| สถานะ | ต้องมี | ตำแหน่งในวงจรชีวิต |
| ระดับความน่าเชื่อถือ | ต้องมี | ระดับ A–E ตามเอกสารแม่ |
| แหล่งข้อมูลหลัก | ต้องมี | ต้นทางสูงสุดที่ถือเป็นต้นฉบับ |
| คำอธิบาย | ควรมี | คำอธิบายขององค์ความรู้ (จำเป็นก่อนเผยแพร่) |
| การอ้างอิง | ควรมี | รายการอ้างอิงที่สืบย้อนได้ (จำเป็นก่อนเผยแพร่) |
| ความสัมพันธ์ | ควรมี | การเชื่อมโยงไปยังองค์ความรู้อื่น |
| ชื่อบาลี | ควรมี | ชื่อในภาษาบาลี (ถ้ามี) |
| เวอร์ชัน | ควรมี | รุ่นปัจจุบันขององค์ความรู้ |
| วันที่สร้าง | ควรมี | วันที่สร้างองค์ความรู้ |
| วันที่ปรับปรุง | ควรมี | วันที่ปรับปรุงล่าสุด |
| ชื่อภาษาอังกฤษ | เพิ่มเติม | ชื่อในภาษาอังกฤษ (ถ้ามี) |
| ชื่อเรียกอื่น | เพิ่มเติม | ชื่อพ้อง/ชื่อที่ใช้เรียกแบบอื่น |
| ข้อมูลกำกับ | เพิ่มเติม | ข้อมูลประกอบอื่น ๆ ที่อธิบายองค์ความรู้ |
| หมายเหตุ | เพิ่มเติม | บันทึกเพิ่มเติมสำหรับผู้ดูแล |

## 6. ข้อมูลที่ต้องมี (Required)

องค์ประกอบที่ **องค์ความรู้ทุกรายการต้องมี** ตั้งแต่แรกสร้าง

| องค์ประกอบ | เหตุผลที่จำเป็น |
|---|---|
| รหัส | ใช้ระบุตัวตนเฉพาะ — ไม่มีรหัสคือไม่มีองค์ความรู้ |
| ชื่อ | ใช้เรียกและอ้างถึงองค์ความรู้ |
| ประเภท | กำหนดว่าองค์ความรู้เป็นชนิดใดตามแบบจำลองโดเมน |
| สถานะ | บอกว่าองค์ความรู้อยู่ ณ ขั้นใดของวงจรชีวิต |
| ระดับความน่าเชื่อถือ | จัดตำแหน่งในลำดับชั้นแหล่งข้อมูล |
| แหล่งข้อมูลหลัก | หลักประกันการตรวจสอบย้อนกลับ |

---

## 7. ข้อมูลที่ควรมี (Recommended)

องค์ประกอบที่ **ควรมีเพื่อความสมบูรณ์** และ **จำเป็นก่อนการเผยแพร่**

| องค์ประกอบ | เหตุผลที่ควรมี |
|---|---|
| คำอธิบาย | ช่วยให้ผู้ใช้เข้าใจองค์ความรู้ |
| การอ้างอิง | ยืนยันความน่าเชื่อถือและการสืบย้อน |
| ความสัมพันธ์ | แสดงการเชื่อมโยงกับองค์ความรู้อื่น |
| ชื่อบาลี | รองรับการศึกษาที่อิงภาษาต้นทาง |
| เวอร์ชัน / วันที่สร้าง / วันที่ปรับปรุง | บันทึกร่องรอยการเปลี่ยนแปลง |

---

## 8. ข้อมูลเพิ่มเติม (Optional)

องค์ประกอบที่ **เพิ่มได้ตามความเหมาะสม** ไม่บังคับ

| องค์ประกอบ | ประโยชน์ |
|---|---|
| ชื่อภาษาอังกฤษ | ช่วยผู้ศึกษาที่ใช้ภาษาอังกฤษ |
| ชื่อเรียกอื่น | รองรับการค้นหาด้วยชื่อพ้อง |
| ข้อมูลกำกับ | เสริมบริบทและการจัดการ |
| หมายเหตุ | บันทึกภายในสำหรับผู้ดูแล |

## 9. กฎการกำหนดรหัส

1. รหัสต้อง **ไม่ซ้ำกัน** ทั้งระบบ และระบุองค์ความรู้ได้เฉพาะหนึ่งเดียว
2. รหัสต้อง **คงที่ (ไม่เปลี่ยนแปลง)** ตลอดอายุขององค์ความรู้
3. **ห้ามนำรหัสที่เคยใช้กลับมาใช้ซ้ำ** แม้องค์ความรู้นั้นถูกยกเลิก
4. รหัสควร **เป็นกลางต่อความหมาย** — ไม่ผูกกับเนื้อหาจนทำให้ต้องเปลี่ยนรหัสเมื่อเนื้อหาเปลี่ยน
5. รูปแบบเชิงเทคนิคของรหัสจะกำหนดในขั้นถัดไป — เอกสารนี้กำหนดเพียง *หลักการ*

---

## 10. กฎการกำหนดชื่อ

1. องค์ความรู้แต่ละรายการมี **ชื่อหลักภาษาไทยเพียงหนึ่งชื่อ** (Canonical Name)
2. ชื่อหลักต้อง **ชัดเจนและสื่อความหมาย** ไม่กำกวม
3. ชื่ออื่น ๆ (บาลี อังกฤษ ชื่อพ้อง) เป็นชื่อรอง ไม่ใช่ชื่อหลัก
4. การเปลี่ยนชื่อหลักต้องมีร่องรอยและเหตุผลกำกับ
5. ชื่อต้องไม่ถูกใช้แทนรหัสในการอ้างอิงเชิงระบบ

---

## 11. หลักการกำหนดชื่อเรียกอื่น

1. ชื่อเรียกอื่น (ชื่อพ้อง / Alias) ทุกชื่อต้อง **เชื่อมโยงกลับไปยังชื่อหลัก** เสมอ
2. ชื่อเรียกอื่นแต่ละชื่อ **ควรมีที่มา** ระบุได้ว่ามาจากแหล่งใด
3. ชื่อเรียกอื่นใช้เพื่อ **ช่วยการค้นหา** ไม่ใช่เพื่อสร้างองค์ความรู้ซ้ำ
4. ชื่อเรียกอื่นต้องไม่ทำให้เกิดความสับสนกับชื่อหลักขององค์ความรู้อื่น

---

## 12. หลักการกำหนดคำอธิบาย

1. คำอธิบายต้อง **แยกชัดเจน** ระหว่างเนื้อหาจากแหล่งข้อมูลหลักกับคำอธิบายของโครงการ
2. ส่วนที่เป็นคำอธิบายของโครงการต้อง **ระบุระดับความน่าเชื่อถือ** (ระดับ D ตามเอกสารแม่)
3. คำอธิบายทุกส่วนที่อ้างเนื้อหาต้นทางต้อง **มีการอ้างอิง**
4. คำอธิบายต้อง **ไม่บิดเบือน ไม่เพิ่มหลักธรรมใหม่** และไม่ตัดทอนจนความหมายเปลี่ยน
5. คำอธิบายต้องเป็นภาษาไทยเป็นหลัก

---

## 13. หลักการเชื่อมโยงการอ้างอิง

1. องค์ความรู้ทุกรายการต้อง **สืบย้อนกลับไปยังแหล่งข้อมูลหลัก** ได้ ตาม [PKA-STD-003](../standards/PKA-STD-003-มาตรฐานการอ้างอิง.md)
2. การอ้างอิงต้อง **แม่นยำถึงระดับที่ผู้ใช้ตรวจสอบได้จริง**
3. ต้องแยก **การอ้างอิงโดยตรง** (เนื้อหาต้นทาง) ออกจาก **การอ้างอิงโดยอ้อม** (คำอธิบาย)
4. องค์ความรู้ที่ยังไม่มีการอ้างอิงที่ตรวจสอบได้ **ห้ามเลื่อนสถานะเป็นเผยแพร่**
5. รายละเอียดรูปแบบการอ้างอิงเป็นไปตาม PKA-STD-003 — เอกสารนี้กำหนดเพียงว่าแม่แบบ *ต้องรองรับ*

---

## 14. หลักการเชื่อมโยงความสัมพันธ์

1. ความสัมพันธ์ทุกเส้นต้องเป็น **ชนิดที่กำหนดไว้ใน [PKA-DOM-001 §6](PKA-DOM-001-แบบจำลองโดเมนองค์ความรู้พระธรรม.md)** (R-1 ถึง R-5)
2. ความสัมพันธ์ทุกเส้นต้อง **มีหลักฐานรองรับ** และสืบที่มาได้
3. **ห้ามสร้างความสัมพันธ์โดยไม่มีหลักฐาน**
4. ความสัมพันธ์อาจมี **ข้อมูลกำกับ** ของตนเอง เช่น ที่มาและระดับความน่าเชื่อถือ
5. ความสัมพันธ์ต้องเชื่อมระหว่างองค์ความรู้ที่มีรหัสจริงเท่านั้น

---

## 15. หลักการกำหนดสถานะการตรวจสอบ

1. องค์ความรู้ทุกรายการต้องมี **สถานะ** ที่บอกตำแหน่งในวงจรชีวิต (ดู §16)
2. การเลื่อนสถานะต้อง **มีผู้รับผิดชอบ** และร่องรอยว่าใครเลื่อนเมื่อใด
3. การเลื่อนสู่ "ผ่านการตรวจสอบ" ต้องมี **ผู้ตรวจสอบ** ที่ระบุได้
4. สถานะต้องสอดคล้องกับ **ระดับความสมบูรณ์** (ดู §17) และผลการตรวจสอบคุณภาพ (ดู §18)
5. การถอยสถานะ (เช่น จากเผยแพร่กลับสู่ตรวจสอบ) ต้องมีเหตุผลกำกับ

---

## 16. วงจรชีวิตขององค์ความรู้

องค์ความรู้แต่ละรายการเดินทางผ่านสถานะตามลำดับนี้

```
   ร่าง
     ▼
   กำลังตรวจสอบ
     ▼
   ผ่านการตรวจสอบ
     ▼
   เผยแพร่
     ▼
   ยกเลิก
```

| สถานะ | ความหมาย |
|---|---|
| **ร่าง (Draft)** | เพิ่งสร้าง มีข้อมูลขั้นต้น ยังไม่ผ่านการตรวจสอบ ยังแก้ไขได้อิสระ |
| **กำลังตรวจสอบ (In Review)** | อยู่ระหว่างการทบทวนความถูกต้องและความครบถ้วน |
| **ผ่านการตรวจสอบ (Verified)** | ผ่านการทบทวนแล้ว มีผู้ตรวจสอบรับรอง แต่ยังไม่เปิดสู่ผู้ใช้ |
| **เผยแพร่ (Published)** | เปิดให้ผู้ใช้เข้าถึง — ต้องมีการอ้างอิงครบและตรวจสอบย้อนกลับได้ |
| **ยกเลิก (Retired)** | ไม่ใช้งานแล้ว แต่คงไว้เพื่อการอ้างอิงและร่องรอย ไม่ลบทิ้ง |

หลักการ:

- การเลื่อนสถานะเป็นไปตามลำดับ การถอยกลับต้องมีเหตุผลกำกับ
- สถานะ "ยกเลิก" **ไม่ลบ** องค์ความรู้ออกจากระบบ เพื่อรักษาร่องรอยและการสืบย้อน

---

## 17. ระดับความสมบูรณ์ขององค์ความรู้

ใช้วัด **ความครบถ้วน** ขององค์ความรู้แต่ละรายการ เป็นคนละมิติกับสถานะ (วงจรชีวิต)

| ระดับ | ชื่อ | เกณฑ์ |
|---|---|---|
| **Level 1** | มีชื่อ | มีรหัสและชื่อหลัก แต่ยังไม่มีคำอธิบาย |
| **Level 2** | มีคำอธิบาย | มีคำอธิบายที่อ่านเข้าใจได้ เพิ่มจาก Level 1 |
| **Level 3** | มีการอ้างอิง | มีการอ้างอิงที่สืบย้อนไปยังแหล่งข้อมูลหลักได้ เพิ่มจาก Level 2 |
| **Level 4** | มีความสัมพันธ์ | มีการเชื่อมโยงกับองค์ความรู้อื่นที่มีหลักฐาน เพิ่มจาก Level 3 |
| **Level 5** | สมบูรณ์ | ครบทุกองค์ประกอบที่ควรมี ผ่านการตรวจสอบคุณภาพ และพร้อมเผยแพร่ |

หลักการ:

- ระดับความสมบูรณ์ **สะสมขึ้นตามลำดับ** — ระดับสูงต้องผ่านเกณฑ์ของระดับล่างก่อน
- องค์ความรู้ควรไปถึง **Level 3 เป็นอย่างน้อย** ก่อนพิจารณาเผยแพร่ และถึง **Level 5** จึงถือว่าสมบูรณ์

---

## 18. หลักการตรวจสอบคุณภาพ (Checklist)

ก่อนเลื่อนองค์ความรู้สู่สถานะ "ผ่านการตรวจสอบ" และ "เผยแพร่" ให้ตรวจตามรายการต่อไปนี้

- ☑ **มีชื่อ** — มีรหัสและชื่อหลักภาษาไทยที่ชัดเจน
- ☑ **มีคำอธิบาย** — มีคำอธิบายที่แยกเนื้อหาต้นทางออกจากคำอธิบายของโครงการ
- ☑ **มีการอ้างอิง** — มีการอ้างอิงที่สืบย้อนไปยังแหล่งข้อมูลหลักได้
- ☑ **มีผู้ตรวจสอบ** — ระบุผู้ตรวจสอบและวันที่ตรวจสอบได้
- ☑ **มีความสัมพันธ์** — มีการเชื่อมโยงที่มีหลักฐาน (เมื่อมีองค์ความรู้ที่เกี่ยวข้อง)
- ☑ **ตรวจสอบย้อนกลับได้** — สายการสืบทอดข้อมูลครบถ้วนไม่ขาดตอน
- ☑ **ระดับความน่าเชื่อถือถูกต้อง** — กำกับระดับ A–E ตรงตามแหล่งที่มา
- ☑ **ไม่บิดเบือน** — ไม่เพิ่มหลักธรรมใหม่ ไม่ตัดทอนจนความหมายเปลี่ยน

> รายการนี้เป็นเกณฑ์ขั้นต่ำเชิงหลักการ รายละเอียดของกระบวนการตรวจสอบจะกำหนดในเอกสารที่เกี่ยวข้องต่อไป

---

## 19. หลักการกำหนดเวอร์ชัน

1. องค์ความรู้แต่ละรายการมี **เวอร์ชันของตนเอง** แยกจากเวอร์ชันของเอกสาร
2. การเปลี่ยนแปลงเนื้อหาที่มีนัยสำคัญต้อง **เพิ่มเวอร์ชัน** และบันทึกเหตุผล
3. ต้องเก็บ **ร่องรอยการเปลี่ยนแปลง** เพื่อให้สืบได้ว่าองค์ความรู้เปลี่ยนอย่างไรเมื่อใด
4. เวอร์ชันเดิมไม่ถูกลบ เพื่อรักษาการสืบย้อน

---

## 20. หลักการกำหนดข้อมูลกำกับ

1. ข้อมูลกำกับ (Metadata) ใช้ **อธิบายองค์ความรู้** ไม่ใช่ตัวเนื้อหาขององค์ความรู้
2. ข้อมูลกำกับขั้นพื้นฐานควรครอบคลุม **ที่มา ระดับความน่าเชื่อถือ สถานะ และร่องรอยเวลา**
3. ข้อมูลกำกับต้องไม่ถูกใช้แทนการอ้างอิงหรือความสัมพันธ์
4. ข้อมูลกำกับต้องสอดคล้องกับมาตรฐานข้อมูล [PKA-STD-002](../standards/PKA-STD-002-มาตรฐานข้อมูล.md)

---

## 21. หลักการรองรับหลายภาษา

1. **ภาษาไทยเป็นภาษาหลัก** ของชื่อและคำอธิบายเสมอ ตามเอกสารแม่
2. รองรับชื่อในภาษาอื่น (เช่น บาลี อังกฤษ) ในฐานะ **ชื่อรอง** โดยไม่แทนที่ชื่อหลัก
3. เนื้อหาแต่ละภาษาต้อง **แยกออกจากกันชัดเจน** และระบุได้ว่าเป็นภาษาใด
4. การเพิ่มภาษาต้องไม่ทำให้การสืบย้อนกลับไปยังแหล่งข้อมูลหลักขาดตอน
5. โครงสร้างต้อง **เปิดให้เพิ่มภาษาได้ในอนาคต** โดยไม่กระทบองค์ประกอบเดิม

---

## 22. สิ่งที่ยังไม่กำหนด

เอกสารฉบับนี้ **จงใจยังไม่กำหนด** สิ่งต่อไปนี้

- ชื่อฟิลด์จริง โครงสร้างการจัดเก็บ หรือรูปแบบข้อมูลเชิงเทคนิค
- การออกแบบฐานข้อมูล JSON SQL หรือ API
- รูปแบบรหัสที่แท้จริงและกฎการสร้างรหัสเชิงเทคนิค
- เนื้อหาพระธรรม องค์ธรรม และรายการจริงขององค์ความรู้
- กระบวนการตรวจสอบคุณภาพโดยละเอียดและผู้รับผิดชอบเชิงปฏิบัติ
- รายละเอียดของข้อมูลกำกับแต่ละประเภทเกินกว่าระดับหลักการ
